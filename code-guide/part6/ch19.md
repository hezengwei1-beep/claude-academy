# Recipes R01-R05：Git 与代码管理

## R01: 智能 Git 提交（完整版）

### Slash Command

`.claude/commands/commit.md`：

```markdown
查看 git status 和 git diff（包括 staged 和 unstaged）。

分析所有改动，生成 commit message：
- 中文，一句话描述"为什么"改，不是"改了什么文件"
- 如果涉及多个方面，用 bullet points
- 遵循项目的 commit 规范

示例好的 message：
- "修复登录超时后 token 未清除导致的重复请求"
- "优化首页加载速度，图片改为懒加载"

示例差的 message：
- "改了 auth.ts"
- "修复 bug"

先展示 message 让我确认。确认后：
1. git add 相关文件（不要 add 不相关的文件）
2. git commit -m "message"
3. 显示 git log --oneline -1 确认
```

### 执行演示

```
> /commit

分析改动...

检测到以下改动：
- src/auth/middleware.ts: 修改了 token 验证逻辑
- src/auth/middleware.test.ts: 新增了超时测试
- src/config.ts: 修改了 TOKEN_EXPIRY 配置

建议的 commit message：

  修复 JWT token 过期后未正确返回 401 的问题，补充超时场景测试

确认提交？(y/n)
> y

[main a1b2c3d] 修复 JWT token 过期后未正确返回 401 的问题，补充超时场景测试
 3 files changed, 42 insertions(+), 8 deletions(-)
```

### 排障

**问题**：Claude 把不相关的文件也 add 了
**解法**：在 CLAUDE.md 中明确 `禁止使用 git add -A 或 git add .`

## R02: PR 变更日志（完整版）

### Slash Command

`.claude/commands/changelog.md`：

```markdown
基于 git log main...HEAD --oneline 和 git diff main...HEAD --stat：

1. 列出所有 commit
2. 按以下分类整理：
   - ✨ Features（新功能）
   - 🐛 Fixes（修复）
   - ♻️ Refactor（重构）
   - 📝 Docs（文档）
   - 🧪 Tests（测试）
   - 🔧 Config（配置）

3. 每条用一句话总结影响，不是复制 commit message
4. 在最后加上"影响范围"：列出改动涉及的模块

输出 Markdown 格式。
```

### 预期输出

```markdown
## Changelog

### ✨ Features
- 用户设置页面新增暗色模式切换

### 🐛 Fixes
- 修复移动端登录按钮在 iOS Safari 上不响应点击
- 修复分页组件在最后一页时"下一页"按钮未禁用

### ♻️ Refactor
- 认证中间件从 class-based 重构为函数式

### 🧪 Tests
- 补充认证模块的边界测试（token 过期、无效签名）

### 影响范围
- `src/auth/` - 认证模块
- `src/components/Pagination/` - 分页组件
- `src/pages/settings/` - 设置页面
```

## R03: Bug 定位与修复（完整版）

### Slash Command

`.claude/commands/fix-bug.md`：

```markdown
Bug 报告：$ARGUMENTS

请按以下流程处理：

**阶段 1：理解**
- 解析 bug 描述，提取关键信息
- 确定 bug 的类型（UI/逻辑/性能/数据）

**阶段 2：定位**
- 搜索相关代码文件
- 分析可能的原因（列出 2-3 个假设）
- 读取相关文件，验证假设
- 确定根因

**阶段 3：方案**（暂停等我确认）
- 给出修复方案
- 评估风险和影响范围
- 需要我确认后才动手

**阶段 4：修复**（确认后）
- 实现修复
- 编写回归测试
- 运行相关测试确认通过
- 生成 commit message
```

### 执行演示

```
> /fix-bug 用户报告在设置页面修改邮箱后点保存，页面白屏

**阶段 1：理解**
- 类型：UI Bug
- 触发条件：设置页面 → 修改邮箱 → 保存
- 症状：白屏（可能是 JS 错误导致 React 崩溃）

**阶段 2：定位**
搜索 src/pages/settings/ ...

找到 src/pages/settings/ProfileForm.tsx:87
```typescript
const handleSave = async () => {
  const result = await updateUser({ email: newEmail });
  setUser(result.data);  // ← 当 API 返回错误时 result.data 是 undefined
};
```

**根因**：updateUser 失败时没有错误处理，result.data 为 undefined，
传给 setUser 后触发下游组件的 TypeError。

**阶段 3：方案**
```typescript
const handleSave = async () => {
  try {
    const result = await updateUser({ email: newEmail });
    if (result.error) {
      setError(result.error.message);
      return;
    }
    setUser(result.data);
    setSuccess('邮箱已更新');
  } catch (err) {
    setError('保存失败，请重试');
  }
};
```

需要确认吗？
```

## R04: 数据库迁移检查（完整版）

### 详细检查清单

```markdown
检查最新的迁移文件，按以下维度评估：

**1. 可回滚性**
- 有没有 down migration？
- down migration 是否能完全还原 up 的改动？
- 测试：执行 up → down → up 是否幂等？

**2. 锁表风险**
- ALTER TABLE ADD COLUMN：PostgreSQL 加 nullable 列不锁表 ✅，加 NOT NULL 列锁表 ⚠️
- ALTER TABLE ADD INDEX：CREATE INDEX CONCURRENTLY 不锁表 ✅，普通 CREATE INDEX 锁表 ⚠️
- ALTER TABLE ALTER COLUMN TYPE：锁表 ⚠️

**3. 数据安全**
- 有没有 DROP COLUMN/DROP TABLE？
- 是否有 NOT NULL 约束添加到已有数据的列？
- 是否有 UNIQUE 约束添加到可能有重复数据的列？

**4. 性能影响**
- 涉及的表有多少行？（查 pg_stat_user_tables）
- 估算执行时间
- 是否需要在低峰期执行？

**5. 分步建议**
- 大迁移是否应该拆成多步？
- 是否需要先部署代码再跑迁移（或反过来）？

输出格式：
🟢 安全 / 🟡 需注意 / 🔴 危险
```

## R05: 依赖更新分析（完整版）

### 执行步骤

```markdown
1. 运行 npm outdated --json

2. 对每个可更新的包，分析：

   **Patch 更新**（1.2.3 → 1.2.4）
   - 通常安全，可以批量更新
   - 但检查 changelog 确认没有意外的 breaking change

   **Minor 更新**（1.2.3 → 1.3.0）
   - 通常安全，但需要看 changelog
   - 关注 deprecation warnings

   **Major 更新**（1.2.3 → 2.0.0）
   - 一定有 breaking changes
   - 必须读 migration guide
   - 建议单独 PR

3. 输出更新计划：

   **Phase 1（立即执行）**：所有 patch 更新
   **Phase 2（本周内）**：安全的 minor 更新
   **Phase 3（需规划）**：major 更新，每个一个 PR

4. 特别关注：
   - 安全漏洞（npm audit 标记的）→ 优先修复
   - 废弃包 → 寻找替代品
   - peer dependency 冲突 → 标注
```
