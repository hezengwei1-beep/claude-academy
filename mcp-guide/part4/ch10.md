# 第十章 MCP 与 Agent Teams 的组合威力

## 为什么要组合

MCP 让 Claude 连接外部世界。Agent Teams 让多个 Claude 并行协作。组合使用 = **多个 Claude 同时操作多个外部系统**。

```
Agent Teams（多代理协作）
    │
    ├── Teammate A ──→ GitHub MCP Server（代码操作）
    ├── Teammate B ──→ PostgreSQL MCP Server（数据查询）
    └── Teammate C ──→ Slack MCP Server（通知发送）
```

## 实战：自动化发布流程

### 场景

每次发布新版本时需要：
1. 检查 GitHub PR 状态
2. 查数据库确认数据迁移完成
3. 在 Slack 通知团队

### 配置

先确保三个 MCP Server 都已配置好，然后：

```
进入 Delegate Mode。

创建一个 3 人发布验证团队：

1. pr-checker
   - 职责：用 GitHub MCP Server 检查所有 open PR 的状态
   - 确认没有阻塞发布的 PR
   - 检查 CI 是否全绿

2. db-checker
   - 职责：用 PostgreSQL MCP Server 查询数据库
   - 确认最新的迁移已执行
   - 检查关键表的数据一致性

3. notifier
   - 职责：等 pr-checker 和 db-checker 都确认后
   - 用 Slack MCP Server 在 #releases 频道发布通知
   - 通知包含：版本号、PR 列表、数据库状态

任务依赖：notifier 依赖前两人完成。
```

## 实战：多源数据分析

### 场景

综合多个数据源做分析报告。

```
创建一个 3 人分析团队：

1. sales-analyst
   - 用 PostgreSQL Server 查询销售数据库
   - 提取上月销售趋势、Top 产品、用户留存

2. feedback-analyst
   - 用 Notion Server 查询用户反馈数据库
   - 提取主要投诉类别、满意度趋势

3. report-writer
   - 等两个分析师完成后
   - 用 Filesystem Server 将综合报告写入 ~/Reports/monthly.md
   - 用 Slack Server 通知管理层报告已就绪
```

## MCP + Agent Teams 的注意事项

### 1. MCP Server 是共享的

所有 teammates 共享同一组 MCP Servers。这意味着：
- 两个 teammates 可以同时查询同一个数据库
- 写入操作需要协调（避免冲突）
- API 限流可能影响并行度

### 2. 成本叠加

Agent Teams 的 token 成本 × MCP 的 API 调用成本。确保：
- 数据库查询有合理的 LIMIT
- API 调用有缓存策略
- 设置 token budget

### 3. 错误传播

一个 MCP Server 的错误可能影响多个 teammates。建议：
- 每个 teammate 的任务描述中包含错误处理指示
- "如果 GitHub API 不可用，跳过 PR 检查并标记为待确认"

## 最佳实践

1. **按数据源分工**：每个 teammate 负责一个 MCP Server，减少资源竞争
2. **读写分离**：只读操作可以并行，写入操作串行或由专人负责
3. **失败优雅降级**：一个数据源不可用不应阻塞整个流程
4. **结果汇总由 Lead 做**：各 teammate 产出原始数据，Lead 做综合分析
