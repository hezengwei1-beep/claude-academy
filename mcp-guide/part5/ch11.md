# 第十一章 企业 MCP 部署案例

## 案例一：内部知识库集成

### 背景

一家 500 人的技术公司，内部知识分散在 Confluence、Notion、Google Drive 和内部 Wiki 四个平台。工程师找一个技术决策的背景信息平均需要 30 分钟。

### 方案

构建了统一的知识库 MCP Server：

```
Knowledge Base MCP Server
├── Tool: search_knowledge(query) → 跨平台搜索
├── Tool: get_document(uri) → 获取文档内容
├── Resource: kb://recent → 最近更新的文档
└── Resource: kb://popular → 最常访问的文档
```

Server 内部对接四个平台的 API，对 Claude 暴露统一接口。

### 结果

- 信息查找时间从 30 分钟降到 2 分钟
- 新人 onboarding 效率提升 3 倍
- 知识库使用率提升 400%（因为通过 Claude 查比自己搜方便得多）

## 案例二：CRM 集成

### 背景

销售团队需要在准备客户会议时查阅 CRM 数据、最近的沟通记录和合同状态。

### 方案

```
CRM MCP Server
├── Tool: get_customer(id) → 客户基本信息
├── Tool: get_communications(customer_id) → 最近的沟通记录
├── Tool: get_deals(customer_id) → 进行中的交易
├── Tool: get_contracts(customer_id) → 合同状态
└── Prompt: meeting_prep(customer_id) → 会议准备模板
```

### 使用方式

销售人员只需说：
```
帮我准备明天和 Acme Corp 的会议，
用 meeting_prep 模板拉取他们的最新信息。
```

Claude 自动拉取 CRM 数据、沟通记录、合同状态，生成结构化的会议准备文档。

## 案例三：数据管道监控

### 背景

数据团队有 50+ 个 ETL 管道，运行在 Airflow 上。每天早上需要检查哪些管道失败了。

### 方案

```
Airflow MCP Server
├── Tool: list_dag_runs(status) → 列出管道运行状态
├── Tool: get_dag_run_logs(dag_id, run_id) → 获取运行日志
├── Tool: trigger_dag(dag_id) → 重新触发管道
└── Resource: airflow://failures/today → 今天的失败列表
```

### 使用方式

```
查看今天所有失败的管道，
分析错误日志找出根本原因，
对于可以安全重试的管道自动触发重跑，
然后生成一份状态报告发到 Slack #data-team 频道。
```

## 企业部署的关键考虑

### 安全

1. **OAuth 2.1**：MCP 内置 OAuth 授权，支持企业 SSO
2. **最小权限**：每个 Server 只暴露必要的工具
3. **审计日志**：记录所有工具调用
4. **数据脱敏**：敏感字段在 Server 端脱敏后再返回

### 架构选择

| 模式 | 适用场景 |
|------|---------|
| **stdio（本地进程）** | 个人使用、开发环境 |
| **Streamable HTTP（远程服务）** | 团队共享、生产环境 |
| **网关代理模式** | 企业级，统一认证和审计 |

### 运维

- MCP Server 作为微服务部署，独立扩缩
- 健康检查和监控告警
- 版本管理和灰度发布
- 定期安全审计

## 案例四：CI/CD 管道集成

### 背景

DevOps 团队需要在 PR 审查时自动拉取多个系统的信息：GitHub PR 详情、Jira ticket 状态、SonarQube 代码质量报告。

### 方案

构建了 CI 集成 MCP Server：

```
CI MCP Server
├── Tool: get_pr_context(pr_url) → PR 改动、评论、CI 状态
├── Tool: get_jira_ticket(key) → 关联的 Jira ticket 详情
├── Tool: get_sonar_report(project) → 代码质量指标
├── Tool: get_deployment_status(env) → 各环境的部署状态
└── Prompt: release_readiness() → 发布就绪检查模板
```

### 实际使用

```
检查 PR #234 是否可以合并：
1. PR 的 CI 是否全绿
2. 关联的 Jira ticket 是否 Ready for Deploy
3. SonarQube 质量门是否通过
4. staging 环境的部署状态
给出 Go / No-Go 建议。
```

### 结果

- 发布决策从 30 分钟缩短到 3 分钟
- 人为遗漏检查项从月均 5 次降到 0
- 团队可以更自信地做日常发布

## 案例五：客服知识增强

### 背景

客服团队每天处理 200+ 工单，需要查阅产品文档、历史工单、用户账户信息。

### 方案

```
Customer Support MCP Server
├── Tool: search_help_docs(query) → 搜索帮助文档
├── Tool: get_user_account(email) → 用户账户和订阅信息
├── Tool: search_past_tickets(query) → 搜索类似的历史工单
├── Tool: get_known_issues() → 当前已知问题列表
└── Prompt: draft_response(ticket) → 基于上下文生成回复草稿
```

### 使用方式

客服人员收到工单后：
```
工单内容：用户说导出 PDF 功能报错。

帮我处理：
1. 查已知问题列表看是否是已知 bug
2. 搜索帮助文档找相关说明
3. 查看用户账户状态（是否是过期账户）
4. 搜索类似历史工单看之前怎么解决的
5. 基于以上信息生成回复草稿
```

### 结果

- 平均工单处理时间从 15 分钟降到 5 分钟
- 客户满意度从 72% 提升到 89%
- 新客服上手时间从 2 周缩短到 3 天

## 构建企业 MCP Server 的模板

### 通用架构

```typescript
// 企业级 MCP Server 标准模板
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";

const server = new McpServer({
  name: "enterprise-server",
  version: "1.0.0",
});

// 1. 健康检查工具（运维必备）
server.tool("health_check", "检查服务健康状态", {}, async () => ({
  content: [{
    type: "text",
    text: JSON.stringify({
      status: "healthy",
      uptime: process.uptime(),
      memory: process.memoryUsage(),
      timestamp: new Date().toISOString(),
    }),
  }],
}));

// 2. 业务工具（按需定义）
server.tool("your_tool", "描述", { /* params */ }, async (params) => {
  // 输入验证
  // 权限检查
  // 执行操作
  // 审计日志
  // 返回结果
});

// 3. 审计中间件
const originalToolCall = server.tool.bind(server);
// 所有工具调用都记录日志

// 4. 启动
const transport = new StdioServerTransport();
await server.connect(transport);
```

### 审计日志格式

```json
{
  "timestamp": "2026-04-27T10:30:00Z",
  "user": "zhang.san",
  "tool": "get_customer",
  "params": {"id": "C-001"},
  "result_size": 1234,
  "duration_ms": 89,
  "success": true
}
```
