
# 第二十四章 Hooks 与质量门控

## Hooks 系统概述

Hooks 是 Claude Code 的事件驱动机制——当特定事件发生时，自动执行你定义的 shell 命令。

**类比**：就像 Git hooks（pre-commit, post-merge），但触发事件是 Claude Code 的操作而非 Git 操作。

## Agent Teams 相关 Hooks

### TeammateIdle

**触发时机**：teammate 即将进入空闲状态（当前任务完成，没有更多分配的任务）。

**用途**：
- 自动分配新任务
- 检查是否还有待完成的工作
- 记录 teammate 的工作时间

### TaskCreated

**触发时机**：新任务被添加到共享任务列表。

**用途**：
- 验证任务描述是否完整
- 自动分类任务优先级
- 通知相关 teammates

### TaskCompleted

**触发时机**：任务被标记为完成。

**用途**：
- 自动运行与任务相关的测试
- 验证代码质量（lint 检查）
- 更新进度报告

## 配置方式

在 `~/.claude/settings.json` 中配置：

```json
{
  "hooks": {
    "TaskCompleted": [
      {
        "command": "cd $PROJECT_DIR && npm test -- --related $CHANGED_FILES",
        "description": "Run related tests on task completion"
      }
    ],
    "TeammateIdle": [
      {
        "command": "echo 'Teammate $TEAMMATE_NAME is idle'",
        "description": "Log idle teammates"
      }
    ]
  }
}
```

## 质量门控实战

### 门控一：自动测试

每当一个 teammate 完成任务时，自动运行相关测试：

```json
{
  "hooks": {
    "TaskCompleted": [
      {
        "command": "npm test -- --changed",
        "description": "Run tests for changed files"
      }
    ]
  }
}
```

如果测试失败，hook 返回非零退出码，Claude Code 会收到通知并尝试修复。

### 门控二：Lint 检查

```json
{
  "hooks": {
    "TaskCompleted": [
      {
        "command": "npx eslint --fix $CHANGED_FILES && npx prettier --write $CHANGED_FILES",
        "description": "Auto-fix lint and format issues"
      }
    ]
  }
}
```

### 门控三：安全扫描

```json
{
  "hooks": {
    "TaskCompleted": [
      {
        "command": "npx semgrep --config=auto $CHANGED_FILES",
        "description": "Security scan on changed files"
      }
    ]
  }
}
```

### 门控四：类型检查

```json
{
  "hooks": {
    "TaskCompleted": [
      {
        "command": "npx tsc --noEmit",
        "description": "TypeScript type check"
      }
    ]
  }
}
```

## 组合门控流水线

将多个门控组合成一个流水线：

```json
{
  "hooks": {
    "TaskCompleted": [
      {
        "command": "npm run lint && npm run typecheck && npm test -- --changed && npm run security-scan",
        "description": "Full quality gate pipeline"
      }
    ]
  }
}
```

用 `&&` 串联意味着前一个失败后续就不执行，节省时间。

## 自定义 Hooks 的注意事项

### 1. 保持轻量

Hook 在 Claude Code 工作流中同步执行。如果 hook 耗时太长，会阻塞 teammate 的工作。

**经验法则**：单个 hook 执行时间控制在 30 秒以内。

### 2. 幂等性

Hook 可能被多次触发。确保你的 hook 是幂等的——多次执行和一次执行的效果相同。

### 3. 错误处理

Hook 返回非零退出码时，Claude Code 会将其视为错误并报告。确保你的 hook 在正常情况下不会意外返回非零码。

### 4. 环境变量

Hooks 中可以使用以下环境变量：
- `$PROJECT_DIR`：项目根目录
- `$CHANGED_FILES`：本次改动的文件列表
- `$TEAMMATE_NAME`：触发 hook 的 teammate 名称
- `$TASK_ID`：相关的任务 ID

## 质量门控的哲学

质量门控的目标不是阻止 AI 做事，而是**在问题变大之前及早发现**。

好的门控：
- 快速反馈（秒级，不是分钟级）
- 只检查有意义的指标
- 自动修复可以修复的问题
- 清晰报告无法自动修复的问题

坏的门控：
- 耗时太长导致 teammate 空等
- 过度严格导致 teammate 无法推进
- 误报太多导致被忽略
