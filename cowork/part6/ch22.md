
# 第二十二章 成本控制与 Token 优化

## 定价基准（2026）

| 模型 | Input | Output | Thinking |
|------|-------|--------|----------|
| Claude Opus 4.6 | $5/M tokens | $25/M tokens | 按 output 计 |
| Claude Sonnet 4.6 | $3/M tokens | $15/M tokens | 按 output 计 |
| Claude Haiku 4.5 | 更低 | 更低 | - |

> M = 百万 tokens。1M tokens ≈ 750,000 个英文单词，或约 15,000 行代码。

## 实际费用数据

来自企业和社区的真实数据：

| 场景 | 费用 |
|------|------|
| 企业平均（每开发者/天） | ~$13 |
| 企业平均（每开发者/月） | $150-250 |
| 90% 的用户（每天） | <$30 |
| C 编译器项目（16 agents, 2 周） | ~$20,000 |
| 49 agents 并行（2.5 小时） | $8,000-15,000 |
| 6 人团队首月 API | $2,400 |
| 单次多代理代码审查 | $15-25 |

## Agent Teams 的成本乘数

Agent Teams 的成本不是简单的"agents 数量 × 单 agent 成本"，因为每个 teammate 有独立的上下文窗口，且每个 teammate 都会读取 CLAUDE.md 等共享配置。

| 配置 | 成本乘数 |
|------|----------|
| 单个 agent | 1x |
| 3 个 subagents | ~2x |
| 3 人 Agent Teams | ~3x |
| 5 人 Agent Teams | ~5x |
| 3 人 Agent Teams + Plan Mode | ~5-7x |

Plan Mode 下成本更高是因为规划阶段消耗额外的 tokens，加上每个 teammate 提交计划审批也消耗 tokens。

## 七个优化策略

### 策略一：不是所有任务都需要多 agent

这是最重要的"优化"——避免不必要的多 agent 使用。

**判断标准**：如果任务用单 agent 能在 30 分钟内完成，就不要上多 agent。

### 策略二：独立并行任务用 /batch

如果任务之间完全独立、不需要通信，`/batch` 比 Agent Teams 更简单更便宜：
- 没有协调开销
- 没有 Task List 同步开销
- 没有 Mailbox 通信开销

### 策略三：设置 Token Budget

在任务描述中明确：
```
Token 预算：本次任务总 token 不超过 500K。
如果接近预算上限，优先完成最关键的部分，
次要任务标记为"因预算限制未完成"。
```

### 策略四：精简 CLAUDE.md

每个 agent/teammate 启动时都会读取 CLAUDE.md。如果你的 CLAUDE.md 有 10,000 tokens，5 个 teammates 就消耗了 50,000 tokens 仅仅用于读取指令。

**社区共识的最佳实践：CLAUDE.md 控制在 ~2,500 tokens（约一页文本）**。

```markdown
# 好的 CLAUDE.md（精简版）
## 项目概述
TypeScript + React + PostgreSQL 的 SaaS 应用

## 关键规则
- 测试用 vitest，不用 jest
- 样式用 Tailwind，不用 CSS modules
- API 返回 snake_case

## 文件约定
- 组件：src/components/[name]/index.tsx
- API：src/api/[resource]/route.ts
```

```markdown
# 差的 CLAUDE.md（冗长版）
## 项目历史
创建于 2024 年 3 月... [500 字的背景故事]

## 详细技术栈
React 18.2.0，TypeScript 5.3... [每个依赖都列出版本号]

## 代码规范
缩进用 2 空格... [重复了 .eslintrc 中已有的规则]
```

### 策略五：Subagent 用便宜的模型

不是所有 subagent 任务都需要 Opus。简单的搜索、统计、格式转换用 Haiku 足够：

```bash
export CLAUDE_CODE_SUBAGENT_MODEL=haiku
```

或者在任务中指定：
```
用 sonnet 模型创建一个 subagent 来搜索代码中的所有 TODO 注释。
```

### 策略六：定期监控用量

养成每周检查 token 使用量的习惯。异常的用量增长通常意味着：
- Agent 陷入了循环
- 任务定义不清晰导致反复尝试
- 不必要地使用了多 agent

### 策略七：分段执行长任务

长时间运行的任务会因为上下文增长而越来越贵（每轮对话都需要处理之前所有的上下文）。

**解法**：把 4 小时的长任务拆成 4 个 1 小时的段，每段开始时只提供必要的上下文。

## 成本估算公式

粗略估算一个 Agent Teams 任务的成本：

```
成本 ≈ teammates 数量 × 单 agent 预估成本 × 1.2（协调开销）
```

例如：单 agent 预估 $5 的任务，3 人 Agent Teams 大约 $18。

## 什么时候钱花得值

不要只看成本，要看 **ROI**：

| 场景 | 单 agent 耗时 | 3 人 Teams 耗时 | 成本增加 | 时间节省 |
|------|-------------|----------------|----------|----------|
| 代码审查 | 60 分钟 | 25 分钟 | 3x | 58% |
| 模块重构 | 4 小时 | 1.5 小时 | 3x | 63% |
| 多维度调查 | 2 小时 | 40 分钟 | 3x | 67% |

如果你的时间价值足够高（大多数专业开发者都是），3 倍的 token 成本换来 60%+ 的时间节省是非常值得的。

关键公式：**如果节省的时间 × 你的时薪 > 额外的 token 成本，那就值。**
