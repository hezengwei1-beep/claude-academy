
# 第二十一章 案例四：失败与教训

成功的案例教你什么是可能的，失败的案例教你什么是必须避免的。本章记录四个真实的失败案例，每个案例都付出了实实在在的代价。

## 案例 A：$47,000 的失控事故

### 发生了什么

一家金融服务公司的团队在使用 Claude Code 时，启动了 23 个 subagents 来并行处理一个大型数据迁移任务。

问题是：**没有人在看**。

团队在周五下午启动了任务，然后回家过周末。三天后回来时发现：

- 23 个 subagents 持续运行了约 72 小时
- 多个 subagent 陷入了死循环（修复 A 破坏 B，修复 B 破坏 A）
- 累积的 API 费用：**$47,000**
- 产出的代码质量极差，大部分需要丢弃

### 根本原因

1. **没有设置 token budget** —— 没有自动停止机制
2. **无人监督** —— 周末没有人检查进度
3. **任务定义不清晰** —— subagents 不知道什么时候该停下来
4. **死循环检测缺失** —— 没有发现 agent 在做重复无用功

### 教训

- **永远设置 token budget**——这是硬性要求
- **不要让 agent 无人看管运行超过几个小时**
- **定义明确的退出条件**——"如果 30 分钟内没有进展就停止"
- **设置成本告警**——达到阈值时通知负责人

### 如何避免

```json
// 设置 token 预算（在任务描述中）
{
  "maxTokens": 1000000,
  "maxDuration": "2h",
  "alertThreshold": "$50"
}
```

在任务描述中明确：
```
如果在 30 分钟内运行了 3 次以上相同的测试仍然失败，
停止工作并报告当前状态，不要继续尝试。
```

## 案例 B：49 代理并行灾难

### 发生了什么

一个开发者编写了一个 "swarm orchestration" 的 slash command，一次性启动了 49 个 subagents 并行工作。

- 49 个 subagent 并行运行了约 2.5 小时
- 估计消耗 API 费用 **$8,000-15,000**
- 多个 subagent 修改了相同的文件，导致大量冲突
- 最终产出大部分不可用

### 根本原因

1. **规模过大** —— 49 个 agent 远超合理范围
2. **没有文件边界** —— 没有约定谁改什么文件
3. **协调机制缺失** —— 49 个 subagent 之间无法通信
4. **好奇心驱动而非需求驱动** —— 为了试试能跑多少而不是解决实际问题

### 教训

- **3-5 个 agents 是经过验证的甜区**
- **Agent 数量不是越多越好**——协调成本随数量非线性增长
- **先在小规模验证，再考虑扩大**
- **用 Agent Teams（有通信能力）而不是纯 subagents（无通信能力）处理需要协调的任务**

## 案例 C：静默性能漂移

### 发生了什么

Anthropic 自己的 Claude Code 系统在一次更新后出现了一个微妙的问题：

- Prompt cache 回归导致 thinking blocks 每小时被清空
- Session 历史在长时间运行后丢失
- 基准测试分数下降了 **3%**

问题的诡异之处在于：**短会话完全正常**。只有在长时间运行的会话（1 小时以上）中才能观察到退化。

### 根本原因

这是一个基础设施层面的缓存回归。Prompt cache 的 TTL（生存时间）配置被意外修改，导致每小时清空一次，而正常应该保持更长时间。

长会话中，每次 cache 失效后 Claude 需要重新处理完整的上下文，但由于某种原因，thinking blocks 在缓存重建后被截断，导致推理能力退化。

### 教训

1. **性能退化可能是渐进的、不易察觉的** —— 如果你觉得 "Claude 最近好像没以前好用了"，可能真的有问题
2. **长会话比短会话更脆弱** —— 定期开新会话而不是一直累积
3. **需要端到端的监控** —— 不只是"能不能跑"，还要监控"跑得好不好"

### 如何避免

- 对于重要的多代理任务，定期检查输出质量
- 如果会话运行超过 1 小时，考虑分段执行
- 保持 Claude Code 更新到最新版本

## 案例 D：盲目信任 —— "The Claude Code Leak"

### 发生了什么

一个开发者让 Claude Code 重构一个包含 API 密钥处理逻辑的模块。Claude 的改动看起来完全正确：代码整洁、逻辑清晰、测试通过。

但问题是：重构后的代码在一个边界情况下，会在日志中明文输出 API 密钥。

这个 bug 通过了代码审查（人工审查者被"看起来很干净"的代码迷惑），通过了所有测试（没有测试覆盖日志输出的内容），进入了生产环境。

### 根本原因

1. **测试覆盖有盲区** —— 没有测试检查日志输出是否包含敏感信息
2. **代码"看起来正确"不等于"真的正确"** —— 整洁的代码更容易让审查者放松警惕
3. **AI 不理解安全的上下文** —— Claude 不知道那个字符串是 API 密钥，它只看到一个需要被日志记录的变量

### 教训

- **AI 生成的代码需要与人写的代码同等甚至更严格的审查**
- **特别关注安全相关的边界情况** —— 敏感数据处理、认证逻辑、权限控制
- **自动化安全扫描是必须的** —— 不能只靠人工审查

### 如何避免

- 在 CLAUDE.md 中明确安全规则：
  ```
  安全规则：
  - 永远不要在日志中输出任何可能包含密钥、token、密码的变量
  - API 密钥必须通过环境变量获取，不能硬编码
  - 日志输出前必须经过脱敏处理
  ```
- 在 CI 中集成安全扫描工具（如 Semgrep、Snyk）
- 对安全关键代码设置额外的审查流程

## 五条黄金法则

综合所有失败案例，提炼出五条黄金法则：

### 法则一：你是项目经理，不是甩手掌柜

Agent Teams 是 **supervised workflow**。你需要：
- 定期检查进度
- 审查输出质量
- 做方向决策
- 及时叫停偏离目标的工作

### 法则二：设置 budget 和告警

**永远**设置：
- Token 预算上限
- 运行时间上限
- 成本告警阈值
- 退出条件（"如果 X 分钟无进展则停止"）

### 法则三：测试通过 ≠ 任务完成

测试只能验证你想到的场景。需要额外关注：
- 安全边界情况
- 性能退化
- 日志和可观察性
- 接口兼容性

### 法则四：从小规模开始

```
2 agents → 验证有效 → 3-5 agents → 验证有效 → 考虑更多
```

不要一开始就上 49 个 agents。

### 法则五：保持更新，保持监控

- Claude Code 版本保持最新
- 长会话定期重开
- 建立产出质量的基线，定期对比

记住这五条法则，你就能避免本章中所有案例的陷阱。
