
# 第二章 MCP 协议深度解析

## 协议基础：JSON-RPC 2.0

MCP 基于 JSON-RPC 2.0 构建。每一次 Client 和 Server 之间的通信都是一个 JSON-RPC 消息。

### 请求格式

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "query_database",
    "arguments": {
      "sql": "SELECT * FROM users LIMIT 10"
    }
  }
}
```

### 响应格式

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "| id | name | email |\n|---|---|---|\n| 1 | Alice | alice@example.com |"
      }
    ]
  }
}
```

## 传输层（Transport）

MCP 支持两种传输方式：

### stdio（标准输入/输出）

Server 作为子进程启动，通过 stdin/stdout 通信。

```
Host 进程 ←→ stdin/stdout ←→ Server 子进程
```

**优点**：简单、零网络配置、天然安全（本地进程间通信）
**缺点**：只能本地使用，不能远程

**适用场景**：本地开发、个人使用、Claude Desktop/Code 的默认模式

### SSE（Server-Sent Events over HTTP）

Server 作为 HTTP 服务运行，通过 HTTP + SSE 通信。

```
Host ←→ HTTP POST（请求）+ SSE（响应流） ←→ Server（HTTP 服务）
```

**优点**：支持远程连接、可以部署在服务器上
**缺点**：需要网络配置、需要考虑安全性

**适用场景**：团队共享 Server、企业部署、远程访问

## 连接生命周期

```
1. 初始化（Initialize）
   Client → Server: initialize 请求（包含 Client 能力）
   Server → Client: initialize 响应（包含 Server 能力）

2. 能力发现（Discovery）
   Client → Server: tools/list    → 获取可用工具列表
   Client → Server: resources/list → 获取可用资源列表
   Client → Server: prompts/list   → 获取可用模板列表

3. 正常通信
   Client ↔ Server: 工具调用、资源读取、通知等

4. 关闭
   Client → Server: shutdown 通知
```

### 初始化握手

```json
// Client → Server
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2024-11-05",
    "capabilities": {
      "roots": { "listChanged": true }
    },
    "clientInfo": {
      "name": "claude-code",
      "version": "2.1.119"
    }
  }
}

// Server → Client
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "protocolVersion": "2024-11-05",
    "capabilities": {
      "tools": {},
      "resources": {},
      "prompts": {}
    },
    "serverInfo": {
      "name": "github-mcp-server",
      "version": "1.0.0"
    }
  }
}
```

## 三种能力详解

### Tools（工具）

工具是 AI 可以调用的函数。每个工具有：
- **名称**：如 `create_issue`
- **描述**：自然语言说明（AI 用此判断何时调用）
- **输入参数 Schema**：JSON Schema 格式
- **返回值**：文本或结构化数据

```json
{
  "name": "create_issue",
  "description": "在 GitHub 仓库中创建一个新 Issue",
  "inputSchema": {
    "type": "object",
    "properties": {
      "repo": { "type": "string", "description": "仓库名，格式 owner/repo" },
      "title": { "type": "string", "description": "Issue 标题" },
      "body": { "type": "string", "description": "Issue 正文" }
    },
    "required": ["repo", "title"]
  }
}
```

AI 看到这个工具描述后，就知道什么时候该调用它、需要传什么参数。

### Resources（资源）

资源是 AI 可以读取的数据。每个资源有 URI 标识：

```json
{
  "uri": "github://anthropics/claude-code/README.md",
  "name": "claude-code README",
  "mimeType": "text/markdown",
  "description": "Claude Code 项目的 README 文件"
}
```

资源支持两种访问模式：
- **直接读取**：`resources/read` 获取内容
- **订阅变更**：`resources/subscribe` 监听更新

### Prompts（提示词模板）

预定义的交互模板，带参数：

```json
{
  "name": "summarize_pr",
  "description": "总结一个 GitHub PR 的改动",
  "arguments": [
    {
      "name": "pr_url",
      "description": "PR 的 URL",
      "required": true
    }
  ]
}
```

## 安全模型

### 权限层级

```
用户授权
  │
  ├── Host 级别：用户决定安装哪些 MCP Server
  │
  ├── Server 级别：每个 Server 声明自己需要的能力
  │
  └── 操作级别：每次工具调用都可以要求用户确认
```

### 人在回路（Human-in-the-Loop）

MCP 的安全设计核心：**敏感操作需要用户确认**。

Host 可以配置：
- 哪些工具自动允许（如只读查询）
- 哪些工具需要用户逐次确认（如写入操作）
- 哪些工具完全禁止

### 数据隔离

每个 MCP Server 是独立进程，只能访问自己被授权的数据。GitHub Server 无法访问 PostgreSQL Server 的数据，反之亦然。

## 与其他方案的对比

| 维度 | MCP | OpenAI Function Calling | LangChain Tools |
|------|-----|------------------------|-----------------|
| 标准化 | 开放协议 | 私有 API | 框架内抽象 |
| 进程隔离 | 独立进程 | 同进程 | 同进程 |
| 传输方式 | stdio / SSE | HTTP API | 函数调用 |
| 安全模型 | 协议级权限 | API Key | 代码级 |
| 生态 | 跨厂商、跨工具 | OpenAI 生态 | Python 生态 |
| 自描述 | Server 自动暴露能力 | 需要手动定义 | 需要手动注册 |

MCP 的核心优势：**跨厂商、进程隔离、自描述、协议级安全**。
