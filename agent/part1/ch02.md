# 第二章 Agent 的工作原理：ReAct 循环

## 所有 Agent 的底层逻辑：ReAct

不管是 Claude Code、ChatGPT Agent、还是任何 Agent 框架，底层都是同一个模式——**ReAct（Reasoning + Acting）**：

```
                    ┌──────────┐
                    │  用户任务  │
                    └────┬─────┘
                         │
                    ┌────▼─────┐
               ┌───►│  Reason  │ ← 思考：当前状态？下一步？
               │    │  (推理)  │
               │    └────┬─────┘
               │         │
               │    ┌────▼─────┐
               │    │   Act    │ ← 行动：调用工具
               │    │  (执行)  │
               │    └────┬─────┘
               │         │
               │    ┌────▼──────┐
               │    │  Observe  │ ← 观察：工具返回了什么？
               │    │  (观察)   │
               │    └────┬──────┘
               │         │
               │    ┌────▼──────┐
               │    │  完成了？  │
               │    └────┬──────┘
               │    No   │   Yes
               └─────────┘    │
                         ┌────▼─────┐
                         │  返回结果  │
                         └──────────┘
```

## 实际运行示例

让我们跟踪一个真实的 Agent 循环：

**用户任务**：`找出 src/auth/ 中的安全问题并修复`

### 循环 1

```
[Reason] 我需要先了解 src/auth/ 目录有哪些文件
[Act]    调用 Glob 工具：pattern="src/auth/**/*.ts"
[Observe] 返回：
  - src/auth/middleware.ts
  - src/auth/login.ts
  - src/auth/token.ts
  - src/auth/types.ts
[判断]   还没完成，需要读取文件内容
```

### 循环 2

```
[Reason] 先读取最关键的 middleware.ts，这通常包含认证逻辑
[Act]    调用 Read 工具：path="src/auth/middleware.ts"
[Observe] 返回文件内容，发现第 42 行：
  const JWT_SECRET = "hardcoded-secret-key-123";
[判断]   发现问题！但需要继续检查其他文件
```

### 循环 3

```
[Reason] 继续检查 login.ts
[Act]    调用 Read 工具：path="src/auth/login.ts"
[Observe] 返回文件内容，发现：
  - 没有 rate limiting
  - 错误消息泄露了用户是否存在（"User not found" vs "Wrong password"）
[判断]   发现更多问题
```

### 循环 4

```
[Reason] 发现了 3 个安全问题，现在开始修复第一个（最严重的）
[Act]    调用 Edit 工具：修改 middleware.ts
  - const JWT_SECRET = "hardcoded-secret-key-123";
  + const JWT_SECRET = process.env.JWT_SECRET!;
[Observe] 文件修改成功
[判断]   第一个修复完成，继续修复第二个
```

### 循环 5-7

```
（修复剩余问题...）
```

### 循环 8

```
[Reason] 所有修复完成，需要验证
[Act]    调用 Bash 工具：npm test
[Observe] 所有测试通过
[判断]   ✅ 任务完成
```

**向用户报告**：
```
发现并修复了 3 个安全问题：
1. [Critical] JWT secret 硬编码 → 改为环境变量
2. [High] 登录无 rate limiting → 添加了限流中间件
3. [Medium] 错误消息泄露用户存在 → 统一为"认证失败"

所有测试通过。
```

## Tool Use：Agent 如何调用工具

### Claude API 层面

在 API 层面，工具调用是这样的：

```
用户消息 → Claude API
Claude 返回：
{
  "role": "assistant",
  "content": [
    {
      "type": "tool_use",       ← Claude 决定调用工具
      "id": "toolu_123",
      "name": "read_file",      ← 工具名
      "input": {                ← 工具参数
        "path": "src/auth/middleware.ts"
      }
    }
  ]
}

应用执行工具，把结果返回：
{
  "role": "tool",
  "tool_use_id": "toolu_123",
  "content": "文件内容..."     ← 工具结果
}

Claude 基于工具结果继续思考...
```

### Claude Code 层面

Claude Code 把上面的过程封装了。你看到的是：

```
> 找出安全问题

Claude: 让我检查 src/auth/ 目录...

[Read src/auth/middleware.ts]

发现第 42 行有硬编码的 JWT secret...

[Edit src/auth/middleware.ts]
  - const JWT_SECRET = "hardcoded-secret-key-123";
  + const JWT_SECRET = process.env.JWT_SECRET!;

需要确认修改吗？[Allow] [Deny]
```

底层是完全一样的 ReAct 循环，只是 Claude Code 提供了更友好的界面。

## Agent 的四种决策模式

### 模式 1：直接回答

```
[Reason] 用户问了一个知识性问题，不需要工具
[Act]    直接生成文本回答
```

### 模式 2：单工具调用

```
[Reason] 需要一个信息，调用一个工具就够了
[Act]    调用一个工具
[Observe] 得到结果
[Act]    基于结果回答用户
```

### 模式 3：多工具链式调用

```
[Reason] 需要先查再改再验证
[Act]    Read → 分析 → Edit → Bash(test) → 报告
```

### 模式 4：并行工具调用

```
[Reason] 需要同时获取多个信息
[Act]    同时调用：Read(file_a) + Read(file_b) + Grep(pattern)
[Observe] 三个结果同时返回
[Act]    综合分析
```

Claude Code 在实际运行中会根据情况自动选择最优的模式。

## 什么让 Agent 变"聪明"

Agent 的质量取决于三个因素：

### 1. LLM 的能力

Opus > Sonnet > Haiku 在推理和决策上有明显差异：
- **Opus**：复杂推理、多步规划、边界情况
- **Sonnet**：日常任务、标准流程
- **Haiku**：简单决策、分类路由

### 2. 工具的设计

工具的 **名称和描述** 决定了 Agent 能否正确选择工具：

```
// 差的工具描述 → Agent 不知道什么时候用
name: "do_thing"
description: "Does a thing"

// 好的工具描述 → Agent 清楚何时调用
name: "query_user_database"
description: "查询用户数据库。输入 SQL SELECT 语句，返回结果行。
只支持 SELECT，不支持写操作。查询超时 30 秒。"
```

### 3. System Prompt 的质量

```
// 差：Agent 不知道自己该做什么
"你是一个 AI 助手"

// 好：Agent 有清晰的角色和规则
"你是一位安全审查专家。你的任务是审查代码中的安全漏洞。
规则：
1. 只使用 Read 和 Grep 工具，不要修改任何文件
2. 按 OWASP Top 10 逐项检查
3. 每个发现标注严重度（Critical/High/Medium/Low）
4. 不确定的问题标注为"需要人工确认"而非假阳性"
```

## 小结

Agent = LLM + 工具 + ReAct 循环。

理解了这个底层模型，你就能理解 Claude Code 的所有行为——为什么它会自动读文件、为什么它需要确认才能修改、为什么有时候会"转圈"。
