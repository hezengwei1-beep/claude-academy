# 第二十六章 反模式清单与排障指南

## 五大反模式

### 反模式一：CLAUDE.md 过度指定

**症状**：Claude 忽略了你在 CLAUDE.md 中的部分规则，或者表现出"选择性遵守"。

**根因**：CLAUDE.md 太长（>5,000 tokens），Claude 在有限的注意力中无法同等重视所有规则。

**解法**：
- 精简到 ~2,500 tokens
- 只保留"不看代码无法推导"的规则
- 删除 .eslintrc / tsconfig.json / package.json 中已有的信息

### 反模式二：Trust-Then-Verify Gap

**症状**：AI 生成的代码"看起来对"，代码审查也通过了，但在生产环境中出问题。

**根因**：
- 人类审查者被"整洁的代码"麻痹了警惕性
- 测试覆盖有盲区（特别是边界情况和安全相关）
- 代码看起来正确 ≠ 行为正确

**解法**：
- AI 生成的代码需要与人写的代码同等甚至更严格的审查
- 特别关注安全边界、错误处理路径、资源清理
- 补充安全相关的自动化测试
- 使用 Hooks 自动运行安全扫描

### 反模式三：无限探索（Infinite Exploration）

**症状**：给 agent 一个开放式的"调查"指令后，agent 读了几百个文件，上下文被占满，最后给出一个模糊的结论。

**根因**：
- 探索范围没有边界
- 没有明确的退出条件
- 没有用 subagent 隔离探索任务

**解法**：
```
# 差：无���界探索
调查一下这个 bug 是怎么产生的

# 好：有边界的探索
调�� src/auth/middleware.ts 中的 token 验证逻辑，
重点看 verifyToken ���数和它调用的方法。
如果原因不在这两个文件中，停止并告诉我你的初步发现。
```

对于确实需要大范围探索的任务，用 subagent 隔离：
```
用一个 subagent 去搜索整个 src/ 目录中所有调用 getUserById 的地方，
只需要告诉我文件名和行号列表。
```

### 反模式四：Context Amnesia

**症状**：主 agent 委派了 subagent 做调查，但自己又重复做了同样的事情。或者主 agent 忽略了 subagent 返回的关键发现。

**根因**：
- 没有明确告诉主 agent "等 subagent 的结果"
- 主 agent 的指令中没有强调依赖 subagent 的产出

**解法**：
```
1. 派 subagent-A 去调查认证模块
2. 派 subagent-B 去调查数据库查询
3. 等两个 subagent 都完成后，基于他们的发现做综合分析
4. 不要自己去读他们已经调查过的文件
```

社区中有人提出了 "agent-coordination.md" 方案——在项目中放一个协调文件，记录哪些 agent 在做什么，避免重复工作。

### 反模式五：同文件冲突

**症状**：两个 teammates 同时修改了同一个文件，导致 Git 冲突或逻辑不一致。

**根因**：创建 teammates 时没有指定文件边界。

**解法**：参见 Ch14（文件边界与冲突管理）。核心做法：
- spawn 时明确每个 teammate 的文件范围
- 共享文件指定唯一修改者
- 使用 Delegate Mode 让 Lead 专注协调

## FAQ 排障指南

### Q1: Agent Teams 启用了但没有效果

**检查**：
```bash
echo $CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS
# 应输出 1
claude --version
# 应 >= 2.1.32
```

如果环境变量设对了但还是不行，检查 settings.json 格式是否有 JSON 语法错误。

### Q2: Teammate 创建���败

**常见原因**：
- API 配额不足（每个 teammate 需要独立连接）
- 网络不稳���
- Claude Code 版本过旧

**解法**：减少 teammate 数量，检查网络，更新 Claude Code。

### Q3: 任务列表不更新

**原因**：Task List 同步有几秒延迟。

**解法**：按 `Ctrl+T` 手动刷新。如果持续不更新，可能是 teammate 已经异常退出。

### Q4: Teammate 陷入循环

**症状**：teammate 反复修改同一个文件，测试反复通过又失败。

**解法**：
1. `Escape` 中断 teammate
2. 查看它的对话历史，理解循环原因
3. 给它更明确的约束或换一种方法
4. 如果无法修复，关闭这个 teammate，创建新的

### Q5: tmux 模式窗格不显示

**检查**：
- tmux 已安装：`which tmux`
- 不要在已有的 tmux 会话中嵌套启动
- VS Code 终端不支持 tmux 模式，用 in-process

### Q6: Token 消耗异常高

**排查**：
- 检查 CLAUDE.md 是否太长
- 检查是否有 teammate 在做无效循环
- 检查 agent 数量是否过多
- 考虑 subagent 使用 haiku 模型

### Q7: 长会话后 Claude "变笨了"

**原因**：上下文窗口接近上限，或 prompt cache 过期。

**解法**：
- 对于超过 1 小时的会话，考虑重新开始
- 把当前成果保存到文件，开新会话继续
- 保持 Claude Code 更新

### Q8: Cowork 标签不显示

**检查**：
- Claude Desktop 是否最新版
- 是否是付费计划（Pro/Max/Team/Enterprise）
- macOS 是否是 Apple Silicon（Intel 不支持）

### Q9: Dispatch 任务卡住

**原因**：桌面端休眠或 Claude Desktop 关闭。

**解法**：
- 确保电脑不休眠
- 确保 Claude Desktop 保持运行
- 检查网络连接

### Q10: Computer Use 操作不准确

**原因**：Computer Use 依赖截图识别，精度有限。

**解法**：
- 放大界面元素
- 给明确的操作步骤
- 避免需要像素级精度的操作
- 关闭无关窗口减少干扰

## 通用排障流程

遇到问题时，按以下顺序排查：

```
1. 版本检查 → claude --version（是否最新？）
2. 配置检查 → settings.json（格式是否正确？）
3. 环境检查 → 环境变量（是否生效？）
4. 网络检查 → 连接是否稳定？
5. 资源检查 → API 配额、磁盘空间
6. 官方文档 → code.claude.com/docs
7. GitHub Issues → github.com/anthropics/claude-code/issues
8. 社区 → 搜索是否有人遇到相同问题
```
