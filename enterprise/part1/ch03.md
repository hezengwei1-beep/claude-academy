
# 第十九章 案例二：多代理代码审查系统

## 背景

2026 年 4 月，Anthropic 公开了其内部使用的 **Claude Code Review** 系统。这个系统将多代理协作应用于一个所有开发团队都需要的场景——代码审查。

与 Carlini 的 C 编译器实验不同，这是一个在生产环境中持续运行的系统，处理的是 Anthropic 内部每天的真实 PR。

## 核心数据

| 指标 | 部署前 | 部署后 |
|------|--------|--------|
| PR 获得实质性审查的比例 | 16% | **54%** |
| 大型 PR（>1000 行）有发现的比例 | - | **84%** |
| 大型 PR 平均发现问题数 | - | **7.5 个** |
| 误报率 | - | **<1%** |
| 单次审查成本 | - | **$15-25** |
| 单次审查耗时 | - | **~20 分钟** |

最引人注目的是那个 **<1% 的误报率**。传统的静态分析工具误报率通常在 30-60%，开发者很快就会因为噪音太多而忽略所有告警。Claude Code Review 做到了极低的误报率，意味着它报告的每一个问题几乎都是真实的。

## 系统架构

### 触发机制

每当有新的 PR 被创建或更新，系统自动触发审查流程。

### 多代理分工

系统为每个 PR 派出一组 agents，并行从不同角度审查：

```
PR #1234 提交
    │
    ├── Agent A：逻辑正确性审查
    │   └── 检查代码逻辑是否正确，是否有边界情况遗漏
    │
    ├── Agent B：安全性审查
    │   └── 检查 OWASP Top 10、认证授权、输入验证
    │
    ├── Agent C：性能审查
    │   └── 检查 N+1 查询、内存泄漏、不必要的计算
    │
    └── Agent D：代码质量审查
        └── 检查可读性、可维护性、测试覆盖率
```

### 结果过滤

所有 agents 的发现汇总后，经过过滤层：
1. 去重（多个 agent 报告同一问题）
2. 按严重度排序（Critical > High > Medium > Low）
3. 过滤低置信度的发现（减少误报）
4. 生成最终审查报告

### 人工决策保留

**关键设计决策：Claude Code Review 不会批准 PR。**

系统只负责发现问题和提供建议。PR 的批准/拒绝/合并仍然是人类审查者的决定。这个设计反映了一个重要理念：AI 是审查的辅助者，不是替代者。

## 为什么从 16% 到 54%

部署前只有 16% 的 PR 获得实质性审查，这不是因为团队偷懒，而是因为：

1. **时间压力**：人类审查者忙于自己的开发任务
2. **审查疲劳**：大型 PR 需要大量时间和精力
3. **知识局限**：不是每个审查者都精通每个模块

Claude Code Review 解决了这三个问题：
- AI 不会累，不受时间压力影响
- 大型 PR 对 AI 来说只是更多文本
- 多个 agent 从不同角度覆盖知识盲区

## 成本效益分析

### 单次审查成本

```
每次审查成本：$15-25
每次审查耗时：~20 分钟
```

### 与人工对比

假设一个高级工程师的时薪是 $100（含福利）：
- 人工审查一个大型 PR：1-2 小时 = $100-200
- AI 审查：$15-25 + 人工确认 15 分钟 = $40-50

**节省约 60-75% 的审查成本**，同时覆盖率从 16% 提升到 54%。

### 隐性收益

比直接成本节省更重要的是：
- **更早发现 bug**：部署前比部署后修复成本低 10-100 倍
- **知识传播**：审查评论帮助代码作者学习最佳实践
- **一致性**：AI 审查不会因为审查者不同而标准不同

## 可复制的要点

如果你想在自己的团队中搭建类似的系统：

### 第一步：定义审查维度

不要试图让一个 agent 做所有事情。拆分审查维度：
- 安全性
- 性能
- 逻辑正确性
- 测试覆盖率
- 代码风格（可选，通常 linter 就够了）

### 第二步：设置 Hooks

使用 Claude Code 的 Hooks 系统，在 PR 事件触发时自动启动审查：

```json
{
  "hooks": {
    "pr-created": {
      "command": "claude --agent code-reviewer --input pr-diff"
    }
  }
}
```

### 第三步：控制误报

误报是代码审查系统的头号杀手。措施包括：
- 要求 agent 只报告高置信度的发现
- 对每个发现解释原因和建议修复
- 允许开发者标记"非问题"来训练过滤规则

### 第四步：保留人工决策

不要让 AI 自动批准或拒绝 PR。AI 发现问题，人做决策。

## 经验教训

1. **多角度并行审查比单角度深度审查更有价值**——大多数严重 bug 是多个小疏忽的组合
2. **低误报率比高覆盖率更重要**——如果开发者不信任审查结果，系统就没用
3. **自动化不是替代人，是放大人的能力**——AI 处理了大量常规审查，人可以把精力放在需要判断力的决策上
4. **成本在规模化后变得微不足道**——每个 PR $15-25，对一个持续产出 PR 的团队来说是非常值得的投资
