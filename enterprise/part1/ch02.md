# C 编译器案例：可复现的配置与操作

> 本章补充 Ch18 的具体可复现配置，让你能在自己的项目中应用同样的多代理协调策略。

## 核心模式提取

Carlini 的 C 编译器项目有三个可以直接复用的模式：

### 模式 1: 文件锁协调（适用于任何多代理项目）

```
project/
├── current_tasks/           # 任务认领目录
│   ├── agent-01-task.txt    # agent-01 正在做什么
│   └── agent-02-task.txt    # agent-02 正在做什么
├── completed_tasks/         # 已完成任务存档
└── src/                     # 共享代码
```

**认领流程**：
```
1. Agent 检查 current_tasks/ 下有没有未被认领的任务
2. 创建 current_tasks/agent-N-doing-XXX.txt
3. 干活
4. 完成后移动到 completed_tasks/
5. 寻找下一个任务
```

**在 Agent Teams 中的等价实现**：
```
创建团队时，在 CLAUDE.md 或 spawn 指令中定义：

每个 teammate 开始工作前，先在 docs/task-board.md 中标记自己认领了哪个任务。
完成后更新状态为 done。
其他 teammate 只认领标记为 open 的任务。
```

### 模式 2: Oracle 策略（适用于有参照标准的任务）

**原理**：用"已知正确"的版本作为对照，把不可并行的大任务变成可并行的小任务。

**编译器场景**：
```
文件 1-100：用 CCC（我们的编译器）编译
文件 101-500：用 GCC（已知正确）编译
把所有目标文件链接��一起
→ 如果能跑，说明文件 1-100 编译正确
→ 逐步扩大 CCC 的范围
```

**你的项目怎么用**：
```
重构场景：
- 模块 A：用新代码
- 模块 B-Z：用旧代码
- 集成测试通过 → 模块 A 重构正确
- 逐步把 B, C, D... 也换成新代码

API 迁移场景：
- Endpoint 1-10：用新 API
- Endpoint 11-50：用旧 API + 适配层
- 逐步迁移
```

### 模式 3: 专业化分工

**编译器中的角色**：
```
- 功能开发者：实现新功能
- 合并者：消除重复代码
- 优化者：提升性能
- 质量者：改善生成代码的质量
```

**你的项目怎么用**：
```
Agent Teams 创建时：

创建 4 人团队：
1. feature-dev — 实现功能
2. quality-guard — 审查 feature-dev 的产出，发现问题发消息回去
3. perf-optimizer — 功能完成后优化性能
4. test-writer — 持续补充测试

这种角色分工比"每人做一个模块"更有效，
因为它形成了内部的质量闭环。
```

## 完整的多代理项目模板

### CLAUDE.md

```markdown
## 多代理协作规则

### 任务认领
- 开始工作前在 docs/tasks.md 标记 [你的名字] - in progress
- 完成后标记 [你的名字] - done
- 不要认领已经有人在做的任务

### 文件边界
- 每个 teammate 只修改分配给自己的文件
- 共享文件（types.ts, constants.ts）由 lead 指定的人修改
- 修改接口定义前发消息给依赖方

### 质量标准
- 每个改动必须通过 npm test
- 不允许 any 类型
- 不允许 @ts-ignore
- 新代码必须有对应测试
```
