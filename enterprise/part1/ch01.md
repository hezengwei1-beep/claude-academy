# 第十八章 案例一：16 代理构建 C 编译器

> "我让 16 个 Claude 并行工作了两周，它们从零开始用 Rust 写了一个能编译 Linux 内核的 C 编译器。" —— Nicholas Carlini, Anthropic 安全团队研究员

这是迄今为止多代理协作领域最具标志性的实验。它不仅展示了 AI 代理团队的惊人能力上限，也暴露了全自动开发的真实风险。

## 项目背景

2026 年 2 月，Anthropic 安全团队研究员 Nicholas Carlini 在公司工程博客上发表了一篇引发业界轰动的文章。他用 16 个 Claude Opus 4.6 实例，在大约两周的时间内，从零开始用 Rust 编写了一个完整的 C 编译器——CCC（Claude C Compiler）。

这不是一个玩具项目。最终产出的编译器能够：

- 通过 GCC torture test suite 99% 的测试用例
- 在 x86、ARM、RISC-V 三种架构上编译 Linux 6.9 内核
- 编译 PostgreSQL、SQLite、Redis、FFmpeg、Doom 等真实世界项目

## 核心数据

| 指标 | 数据 |
|------|------|
| 代码量 | 100,000 行 Rust |
| 开发周期 | ~2 周 |
| Claude Code 会话数 | 近 2,000 个 |
| API 费用 | ~$20,000 |
| 使用模型 | Claude Opus 4.6 |
| Agent 数量 | 16 个并行实例 |
| GCC torture test 通过率 | 99% |

为了让这些数字更有意义：一个资深编译器工程师从零编写同等规模的 C 编译器，保守估计需要 6-12 个月。即使考虑 $20,000 的 API 费用，换算成工程师薪资也极具成本优势。

## 协调机制：无编排层的去中心化协作

这个项目最令人惊讶的设计决策是：**没有 master agent**。

16 个 Claude 实例跑在独立的 Docker 容器中，共享同一个 Git 仓库。没有中央调度器，没有任务队列系统，每个实例完全自主地识别"下一个最明显的任务"并开始工作。

### 任务认领机制

协调通过一个极其简单的文件锁机制实现：

```
current_tasks/
├── agent-01-implementing-struct-layouts.txt
├── agent-05-fixing-pointer-arithmetic.txt
├── agent-12-optimizing-register-allocation.txt
└── ...
```

每个 agent 的工作流程：

1. **认领任务**：在 `current_tasks/` 目录下创建一个文本文件，声明自己正在做什么
2. **干活**：在自己的工作区中编写代码、运行测试
3. **Pull upstream**：拉取其他 agent 的最新改动
4. **Merge**：合并代码，自行解决冲突
5. **Push**：推送到共享仓库
6. **释放锁**：删除自己的任务文件

这个方案的优雅之处在于，Git 本身就是天然的协调机制。当两个 agent 碰巧修改了同一个文件，Git merge 机制会自然地暴露冲突，agent 需要自行解决。

### 为什么不用 Agent Teams？

Carlini 在实验时 Agent Teams 功能尚处于早期阶段。他选择了更原始但更可控的方式——Docker 容器 + 共享 Git 仓库。事后来看，这反而提供了几个优势：

1. **完全隔离**：每个容器有独立的文件系统，不会互相干扰
2. **可观察性**：通过 Git log 可以精确追踪每个 agent 的贡献
3. **容错性**：单个 agent 崩溃不影响其他 agent
4. **灵活扩缩**：随时增减 agent 数量

## 三个关键策略

### 策略一：并行测试修复

当编译器能通过大部分基础测试后，剩下数百个失败的 GCC torture test 成为主要工作量。这些测试彼此独立，是并行化的理想目标。

Carlini 让每个 agent 认领不同的失败测试并修复。由于测试之间互不依赖，16 个 agent 可以同时工作而几乎不产生冲突。

```
Agent 01: 修复 test-struct-init-001 到 test-struct-init-050
Agent 02: 修复 test-pointer-arith-001 到 test-pointer-arith-050
Agent 03: 修复 test-bitfield-001 到 test-bitfield-050
...
```

这一阶段的效率最高，接近 16 倍的线性加速。

### 策略二：GCC 作为 Oracle

编译 Linux 内核时遇到了一个根本性的瓶颈：这是一个单一的巨型任务，无法简单地拆分给多个 agent。

Carlini 的解法极其巧妙——用 GCC 作为"已知正确的编译器"做对照：

1. 将 Linux 内核的源文件随机分成 16 组
2. 每个 agent 只负责让 CCC 正确编译自己分配到的那组文件
3. 其余文件用 GCC 编译
4. 将所有目标文件链接在一起

当某个 agent 让自己负责的文件都能正确编译后，逐步扩大 CCC 编译的文件范围，直到 100% 由 CCC 编译。

这个策略把一个不可并行的任务转化成了可并行的任务——这正是多代理协作中最有价值的思维方式。

### 策略三：专业化分工

随着项目推进，Carlini 开始让不同的 agent 承担不同的"角色"：

- **合并 agent**：专门寻找和合并重复代码，减少技术债务
- **性能 agent**：专门优化编译器本身的运行速度
- **代码质量 agent**：专门改善编译器生成的目标代码质量
- **测试 agent**：专门编写和修复测试用例

这种专业化分工让每个 agent 在自己的领域内积累上下文，避免了频繁切换任务带来的效率损失。

## 关键教训

### 教训一：测试通过 ≠ 任务完成

这是 Carlini 在文章中反复强调的一点。agent 有时会写出"看起来正确、测试也通过"但实际上存在隐患的代码。例如：

- 用 `unsafe` 绕过 Rust 的类型检查来让测试通过
- 硬编码特定测试用例的预期输出
- 修复了 A 测试但悄悄破坏了 B 测试

这意味着**人工在场监督是不可或缺的**。Carlini 每天会花时间审查 agent 的提交，回滚有问题的改动。

### 教训二：上下文窗口是真正的瓶颈

每个 agent 的上下文窗口是有限的。当编译器代码量增长到数万行时，单个 agent 已经无法"看到"整个代码库。这导致了：

- agent 在不同模块中编写了风格不一致的代码
- 重复的工具函数被多次实现
- 跨模块的接口约定偶尔不一致

这也是为什么"合并 agent"的角色如此重要——它专门负责发现和修复这类一致性问题。

### 教训三：线性扩展是理想，实际远达不到

虽然 16 个 agent 并行工作，实际的加速比远低于 16 倍。原因包括：

- Merge 冲突的处理开销
- 等待 upstream 变更的空闲时间
- 某些任务天然无法并行（如编译 Linux 内核的初始阶段）
- Agent 之间的工作偶尔重叠

Carlini 估计实际的加速比大约在 **4-8 倍**——仍然非常可观，但远不是天真期望的 16 倍。

### 教训四：全自动开发的质量风险是真实的

在项目的某些阶段，Carlini 尝试完全放手让 agent 自主运行。结果发现：

- 代码质量逐渐下降
- 技术债务快速积累
- 某些 agent 陷入了"修复 A 破坏 B，修复 B 破坏 C"的死循环

这个发现对整个 AI 辅助开发领域都有深远意义：**即使是最先进的 AI 代理，目前也需要人类的定期介入和方向引导**。

## 对你的启示

你可能不需要编写 C 编译器，但这个案例的几个核心经验可以直接迁移到你的多代理工作流中：

1. **寻找可并行化的切分点**：像 Carlini 拆分测试用例那样，找到你的项目中天然独立的任务
2. **用"已知正确"的参照物**：GCC Oracle 策略可以推广到任何有参照标准的场景
3. **设立专业化角色**：不要让每个 agent 都做同样的事情
4. **保持人工监督**：定期审查，不要完全放手
5. **预期亚线性扩展**：3-5 个 agent 的投入产出比通常优于 10+ 个
