
# 第六章 场景实战：代码生成与审查

## 代码生成的 Prompt 优化

### 核心原则：约束越明确，代码越好

**差的 prompt**：
```
写一个登录接口
```

**好的 prompt**：
```xml
<task>
实现一个用户登录 API endpoint
</task>

<tech_stack>
- Framework: FastAPI
- 数据库: PostgreSQL + SQLAlchemy
- 认证: JWT (PyJWT)
- 密码: bcrypt
</tech_stack>

<requirements>
1. POST /api/auth/login
2. 请求体: {"email": string, "password": string}
3. 成功返回: {"access_token": string, "token_type": "bearer"}
4. 失败返回: 401 + {"detail": "Invalid credentials"}
5. JWT 有效期: 24 小时
6. 密码验证使用 bcrypt
7. 防止时序攻击
</requirements>

<constraints>
- 不要用 any 类型
- 所有输入要做验证（Pydantic）
- 错误消息不能泄露用户是否存在
- 添加 rate limiting 注释标记
</constraints>

<output>
完整的 Python 文件，包含所有 import
</output>
```

### 迭代优化技巧

1. **第一轮**：生成基础代码
2. **第二轮**：`这段代码有安全问题吗？请修复`
3. **第三轮**：`请补充单元测试`
4. **第四轮**：`请优化性能，特别是数据库查询部分`

每一轮让 Claude 专注一个维度，比一次性要求"写一个完美的代码"效果更好。

## 代码审查的 Prompt 模板

### 通用审查模板

```xml
<role>
你是一位资深代码审查者，有 10 年后端开发经验。
审查风格：直接指出问题，不做不必要的夸奖。
</role>

<code language="typescript">
[待审查的代码]
</code>

<review_dimensions>
1. **安全性**：OWASP Top 10，输入验证，认证授权
2. **性能**：N+1 查询，不必要的计算，内存泄漏
3. **可维护性**：命名，复杂度，耦合度
4. **错误处理**：异常捕获，边界情况，降级策略
5. **测试**：覆盖率，边界测试，mock 合理性
</review_dimensions>

<output_format>
每个问题：
[Critical/High/Medium/Low] 文件:行号
问题描述
建议修复（含代码示例）
</output_format>
```

### 安全审查专用

```xml
<role>
你是一位 Web 安全专家，专注于发现代码中的安全漏洞。
</role>

<task>
审查以下代码中的安全问题，重点关注：
- SQL 注入
- XSS（跨站脚本）
- CSRF（跨站请求伪造）
- 认证绕过
- 敏感信息泄露
- 不安全的直接对象引用
- 密码学误用
</task>

<code>
[代码]
</code>

<output_format>
| 严重度 | 漏洞类型 | 位置 | 描述 | 修复建议 | OWASP 分类 |
</output_format>
```

### PR 审查模板

```
审查以下 Git diff：

<diff>
[git diff 输出]
</diff>

<context>
这是一个 [项目描述] 的 [功能/修复] PR。
背景：[为什么做这个改动]
</context>

请关注：
1. 改动是否符合声称的目的
2. 是否引入了新 bug
3. 是否有遗漏的边界情况
4. 是否需要补充测试
5. 是否有性能退化风险

输出格式：
- 必须修复（Blocking）：
- 建议修改（Suggestion）：
- 赞（Good）：
- 总结：
```

## 代码解释

```xml
<task>
用简单的语言解释以下代码的工作原理。
假设读者了解基本的编程概念但不了解这个项目。
</task>

<code>
[代码]
</code>

<output_format>
1. 一句话概述这段代码做什么
2. 逐步解释核心逻辑（用编号步骤）
3. 关键设计决策和它们的原因
4. 潜在的改进点
</output_format>
```

## 重构建议

```xml
<task>
分析以下代码的"代码异味"（code smells），
提出重构建议。
</task>

<code>
[代码]
</code>

<principles>
参考原则：
- SOLID
- DRY
- KISS
- 关注点分离
</principles>

<output_format>
每个建议：
1. 异味描述
2. 为什么是问题
3. 重构方案（含代码示例）
4. 预期收益
</output_format>
```
