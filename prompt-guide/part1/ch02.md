
# 第二章 Prompt 基础技术

## Prompt 的四个组成部分

```
┌─────────────────────────────────────┐
│ System Prompt（系统提示）             │
│ 定义 Claude 的角色、行为和约束        │
├─────────────────────────────────────┤
│ Context（上下文）                     │
│ 提供背景信息、参考资料                │
├─────────────────────────────────────┤
│ Task（任务）                         │
│ 明确告诉 Claude 要做什么              │
├─────────────────────────────────────┤
│ Output Format（输出格式）             │
│ 指定期望的输出结构                    │
└─────────────────────────────────────┘
```

## System Prompt

System Prompt 是给 Claude 的"角色说明书"，在对话开始前设定。

### 好的 System Prompt

```
你是一位资深的 Python 后端工程师，专注于 Django 和 FastAPI。
你的回答风格：
- 先给结论，再解释原因
- 代码示例使用 type hints
- 安全问题优先提醒
- 回答用中文，代码用英文
```

### 差的 System Prompt

```
你是一个 AI 助手，帮我回答问题。
（太泛，没有任何具体指导）
```

### System Prompt 的权重

Claude 对 System Prompt 的遵循度极高。这意味着：
- 在 System Prompt 中设定的规则，Claude 会严格执行
- 如果 System Prompt 和 User Prompt 有冲突，Claude 倾向于遵循 System Prompt
- 利用这个特性做安全护栏非常有效

## Zero-shot vs Few-shot

### Zero-shot（零样本）

不给示例，直接描述任务：

```
把以下英文翻译成中文，保持专业术语：
"The Model Context Protocol enables AI applications to connect with external data sources."
```

### Few-shot（少样本）

给出几个示例，让 Claude 学习模式：

```
按以下格式分类用户反馈：

示例 1：
输入："登录页面加载太慢了"
输出：{"category": "性能", "severity": "medium", "component": "认证"}

示例 2：
输入："希望能导出 PDF 报告"
输出：{"category": "功能需求", "severity": "low", "component": "报告"}

现在请分类：
输入："支付完成后没有收到确认邮件"
```

### 选择建议

| 场景 | 推荐 |
|------|------|
| 任务简单明确 | Zero-shot |
| 需要特定输出格式 | Few-shot（2-3 个示例） |
| 分类/标签任务 | Few-shot |
| 创意写作 | Zero-shot（避免限制创意） |
| 代码生成 | Zero-shot（Claude 代码能力强） |

## 角色提示（Role Prompting）

给 Claude 一个具体角色，可以显著提升特定领域的输出质量：

```
你是一位有 15 年经验的数据库性能优化专家。
用户会给你 SQL 查询，你需要：
1. 分析查询的性能瓶颈
2. 给出优化方案
3. 解释为什么这个方案更好
4. 评估优化后的预期提升
```

### 角色设计原则

1. **具体化**："15 年经验的数据库专家" 优于 "一个专家"
2. **对齐任务**：角色应该与任务直接相关
3. **设定风格**：告诉 Claude 回答的风格和深度
4. **别演过头**：角色是引导而非束缚，太复杂的角色设定反而降低效果

## 结构化输出

### JSON 格式

```
分析以下用户评论，输出 JSON 格式：

评论："产品质量不错，但发货太慢了，等了一周才收到。"

输出格式：
{
  "sentiment": "mixed/positive/negative",
  "aspects": [
    {"aspect": "string", "sentiment": "string", "detail": "string"}
  ],
  "summary": "string"
}
```

### Markdown 表格

```
对比 React 和 Vue 的状态管理方案，用 Markdown 表格呈现：
列：方案名称、复杂度、学习曲线、适用场景、社区活跃度
```

### 强制格式

如果 Claude 偶尔不遵循格式，可以更强硬：

```
你的输出必须是纯 JSON，不要有任何其他文字。
不要用 markdown 代码块包裹。
直接输出 JSON 对象。
```

## Chain of Thought（思维链）

让 Claude 一步步推理，而不是直接给答案：

```
请一步一步分析这个 bug：
1. 首先，阅读错误信息，理解它在说什么
2. 然后，查看相关代码，定位可能的原因
3. 接着，考虑每个可能原因的概率
4. 最后，给出最可能的根因和修复建议

错误信息：TypeError: Cannot read properties of undefined (reading 'map')
相关代码：...
```

### Chain of Thought vs Extended Thinking

| | Chain of Thought | Extended Thinking |
|---|---|---|
| 方式 | 在 prompt 中要求"一步步思考" | 模型内部自动推理 |
| 可见性 | 推理过程在输出中可见 | thinking blocks 可选可见 |
| 控制 | 你控制推理步骤 | 模型自主决定怎么推理 |
| 成本 | 推理步骤算 output tokens | thinking tokens 算 output 价格 |
| 最佳场景 | 想看到推理过程 | 想要最好的推理质量 |
