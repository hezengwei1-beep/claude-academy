# 第五章 Prompt Caching 与成本优化

## Prompt Caching 是什么

Prompt Caching 是 Claude API 的一个特性：如果你连续多次请求中有相同的前缀部分，这部分只在第一次处理，后续请求直接复用缓存。

```
请求 1: [System Prompt + 大量上下文] + 用户问题 A  → 全价
请求 2: [System Prompt + 大量上下文] + 用户问题 B  → 缓存命中，前缀部分免费
请求 3: [System Prompt + 大量上下文] + 用户问题 C  → 缓存命中
```

### 节省多少

缓存命中的部分按 **input 价格的 10%** 计费。

例如 Opus 4.6:
- 正常 input: $5/M tokens
- 缓存命中: $0.5/M tokens
- **节省 90%**

### 缓存条件

- 前缀必须完全相同（一个字符的差异就不会命中）
- 缓存 TTL 约 5 分钟
- 前缀长度 >= 1024 tokens（太短不缓存）

## 针对 Caching 优化 Prompt 结构

### 原则：不变的放前面，变化的放后面

```
┌────────────────────────────────────┐
│ System Prompt（不变）               │  ← 缓存
├────────────────────────────────────┤
│ 大量参考文档/代码（不变或少变）     │  ← 缓存
├────────────────────────────────────┤
│ Few-shot 示例（不变）              │  ← 缓存
├────────────────────────────────────┤
│ 用户的具体问题（每次不同）          │  ← 不缓存
└────────────────────────────────────┘
```

### 实际示例

**差的结构**（缓存失效）：

```python
# 每次请求都变化的内容放在了前面
messages = [
    {"role": "user", "content": f"今天是 {today}，请分析..."}, 
    {"role": "user", "content": LARGE_CONTEXT},  # 5000 tokens 的上下文
]
```

**好的结构**（缓存命中）：

```python
messages = [
    {"role": "user", "content": [
        {"type": "text", "text": LARGE_CONTEXT, "cache_control": {"type": "ephemeral"}},
        {"type": "text", "text": f"今天是 {today}，请分析..."},
    ]},
]
```

### cache_control 标记

在 API 中，用 `cache_control` 标记缓存断点：

```python
response = client.messages.create(
    model="claude-sonnet-4-6",
    system=[
        {
            "type": "text",
            "text": "你是一个代码审查专家...",  # System prompt
            "cache_control": {"type": "ephemeral"}  # 标记缓存断点
        }
    ],
    messages=[
        {
            "role": "user",
            "content": [
                {
                    "type": "text",
                    "text": LARGE_CODEBASE,  # 大量代码上下文
                    "cache_control": {"type": "ephemeral"}
                },
                {
                    "type": "text",
                    "text": "请审查最近的改动"  # 变化的部分
                }
            ]
        }
    ]
)
```

## 其他成本优化策略

### 1. 模型降级

不是所有任务都需要 Opus：

```python
# 简单分类用 Haiku
classification = client.messages.create(
    model="claude-haiku-4-5",
    messages=[{"role": "user", "content": "分类这个文本..."}]
)

# 复杂分析用 Opus
analysis = client.messages.create(
    model="claude-opus-4-6",
    messages=[{"role": "user", "content": "深度分析这个架构..."}]
)
```

### 2. max_tokens 限制

明确限制输出长度，避免不必要的长输出：

```python
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=500,  # 限制输出
    messages=[{"role": "user", "content": "用一句话总结..."}]
)
```

### 3. Batch API

对于不需要实时响应的任务，用 Batch API 可以享受 50% 折扣：

```python
# 创建 batch
batch = client.batches.create(
    requests=[
        {"custom_id": "req1", "params": {"model": "claude-sonnet-4-6", "messages": [...]}},
        {"custom_id": "req2", "params": {"model": "claude-sonnet-4-6", "messages": [...]}},
        # ... 更多请求
    ]
)

# 异步获取结果（通常在 24 小时内完成）
results = client.batches.results(batch.id)
```

### 4. Streaming 减少感知延迟

Streaming 不省钱，但减少用户等待感：

```python
with client.messages.stream(
    model="claude-sonnet-4-6",
    messages=[...],
) as stream:
    for text in stream.text_stream:
        print(text, end="", flush=True)
```

## 成本监控

### 从 API 响应中读取用量

```python
response = client.messages.create(...)

print(f"Input tokens: {response.usage.input_tokens}")
print(f"Output tokens: {response.usage.output_tokens}")
print(f"Cache read: {response.usage.cache_read_input_tokens}")
print(f"Cache creation: {response.usage.cache_creation_input_tokens}")
```

### 计算实际成本

```python
def calculate_cost(usage, model="claude-sonnet-4-6"):
    prices = {
        "claude-opus-4-6": {"input": 5, "output": 25, "cache_read": 0.5},
        "claude-sonnet-4-6": {"input": 3, "output": 15, "cache_read": 0.3},
    }
    p = prices[model]
    
    cost = (
        usage.input_tokens * p["input"] / 1_000_000 +
        usage.output_tokens * p["output"] / 1_000_000 +
        usage.cache_read_input_tokens * p["cache_read"] / 1_000_000
    )
    return cost
```
