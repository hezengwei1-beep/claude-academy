# 第四章 Extended Thinking 深度运用

## 工作原理

Extended Thinking 让 Claude 在生成最终回答前，先进行一轮内部推理。这个推理过程产生 `thinking` blocks，包含 Claude 的分析、假设、自我纠正等。

```
用户输入 → [thinking: 分析问题...考虑多种可能...排除错误方向...] → 最终回答
```

### API 启用方式

```python
response = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=16000,
    thinking={
        "type": "enabled",
        "budget_tokens": 10000  # thinking 的 token 预算
    },
    messages=[{"role": "user", "content": "分析这个 bug..."}]
)
```

### 在 Claude Code 中

Claude Code 自动根据任务复杂度决定是否使用 thinking。你也可以通过 prompt 暗示：

```
请仔细分析这个问题，考虑所有可能的原因后再给出结论。
```

## 最佳使用场景

### 1. 数学和逻辑推理

```
一个水池有两个进水管和一个排水管。
A管单独进水4小时注满，B管单独进水6小时注满，
排水管单独排水8小时排空。
如果三管同时打开，多久注满水池？

请使用 Extended Thinking 仔细计算。
```

thinking 过程中 Claude 会列出方程、验算、自我检查。

### 2. 复杂代码 Bug 定位

```
以下代码有一个间歇性出现的 race condition。
请仔细分析所有可能的并发场景，
找出 race condition 的精确触发条件。

[代码]
```

### 3. 架构决策

```
我们需要在以下三种方案中做选择：
A. 微服务架构 + Kubernetes
B. 单体应用 + 水平扩展
C. Serverless + 事件驱动

项目背景：[详细描述]

请深入分析每种方案的优劣，
考虑 2 年后的维护成本和 5 年后的扩展性。
```

### 4. 安全审计

```
请深度审查以下认证代码，
考虑所有 OWASP Top 10 的攻击向量，
不要遗漏任何边界情况。

[代码]
```

## 不适合使用的场景

| 场景 | 为什么不需要 |
|------|-------------|
| 简单问答 | "Python 的 list 和 tuple 的区别"——不需要深度推理 |
| 文本翻译 | 翻译是模式匹配，不是推理 |
| 格式转换 | JSON 转 CSV 不需要思考 |
| 信息提取 | 从文本中提取日期/姓名不需要推理 |

## Thinking Budget 管理

### budget_tokens 参数

控制 Claude 可以用多少 tokens 来思考：

| budget_tokens | 适用场景 |
|---------------|---------|
| 2,000-5,000 | 中等复杂度的分析 |
| 5,000-10,000 | 复杂推理、代码审计 |
| 10,000-20,000 | 极其复杂的多步推理 |

### 成本影响

thinking tokens 按 **output 价格**计费：
- Opus 4.6: $25/M tokens
- 10,000 thinking tokens ≈ $0.25

对比直接回答（通常 1,000-3,000 output tokens），使用 thinking 的成本可能高 3-10 倍。

### 优化策略

1. **只在需要时启用**：不是所有任务都需要 thinking
2. **设合理的 budget**：不要给简单任务设 20,000 budget
3. **用 Sonnet 替代 Opus**：如果 Sonnet 的 thinking 质量够用

## Thinking 与 Prompt 的配合

### 引导 thinking 方向

虽然你不能直接控制 thinking 的内容，但可以通过 prompt 暗示方向：

```
分析这段代码时，请特别注意：
1. 线程安全性
2. 内存泄漏
3. 错误处理路径
```

Claude 的 thinking 会优先关注这三个方向。

### 要求展示推理过程

如果你想看到推理过程（而不是只看到最终结论）：

```
请展示你的完整分析过程，包括：
- 你考虑了哪些可能性
- 为什么排除了某些选项
- 最终结论的信度
```

这与 Extended Thinking 不冲突——thinking 是内部推理，这是要求 Claude 在输出中也展示推理。

## 实战对比

### 不用 Thinking

```
问：分析这个死锁问题
答：这里有两个线程互相等待对方的锁，建议用锁排序来解决。
```

### 用 Thinking

```
问：分析这个死锁问题
[thinking: 
  让我分析线程 A 和线程 B 的锁获取顺序...
  线程 A: lock(mutex_1) → lock(mutex_2)
  线程 B: lock(mutex_2) → lock(mutex_1)
  这构成了经典的循环等待...
  但是，还有一个隐藏的问题——线程 C 也在获取 mutex_1...
  结合三个线程的完整顺序分析...
]
答：这里有三个线程参与的死锁，不只是 A 和 B。线程 C 的 mutex_1 
   获取进一步加剧了竞争条件。建议：
   1. 统一所有线程的锁获取顺序为 mutex_1 → mutex_2
   2. 线程 C 应该使用 try_lock 带超时机制
   3. 考虑用读写锁替代互斥锁减少竞争
```

Thinking 帮助 Claude 发现了第三个线程的参与——这在快速回答中很容易被忽略。
