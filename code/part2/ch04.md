# 第四章 对话管理与上下文控制

## 上下文窗口的本质

每次你和 Claude Code 对话，所有的历史消息、读取过的文件内容、命令输出都在"上下文窗口"中累积。Claude Opus 4.6 有 1M tokens 的窗口，但不意味着你可以无限聊。

### 上下文的构成

```
System instructions（CLAUDE.md 等）  ~2,000 tokens
├── 对话历史                          累积增长
│   ├── 你的消息                     
│   ├── Claude 的回复                
│   ├── 文件内容（Claude 读取的）      每个文件 500-30,000 tokens
│   └── 命令输出                      可能很大
└── 当前轮次的思考和输出               
```

### 什么时候上下文不够

症状：
- Claude 开始"忘记"之前讨论的内容
- 回复质量明显下降
- 提示上下文即将压缩

## /compact —— 智能压缩

```
/compact
```

Claude 会：
1. 总结之前的对话关键信息
2. 丢弃不再需要的细节（如已经修复的 bug 的排查过程）
3. 保留当前仍然需要的上下文

**何时用**：
- 对话超过 20 轮后
- 切换到新话题时
- 感觉 Claude "变笨"时

**带自定义指示的压缩**：

```
/compact 保留关于数据库迁移的讨论，其他可以丢弃
```

## /resume —— 会话恢复

Claude Code 会自动保存会话。下次启动时恢复：

```bash
claude --resume
```

或在 Claude Code 中：

```
/resume
```

会列出最近的会话供你选择恢复。

### 最佳实践

- 每个功能/任务用一个会话
- 功能完成后开新会话，不要在同一个会话中混杂多个不相关任务
- 重要的中间成果保存到文件，而不是只存在对话中

## /rewind —— 回滚

```
/rewind
```

回滚到上一个检查点。适用于：
- Claude 的最后一次改动搞坏了什么
- 想尝试另一种方案
- 误操作需要撤回

## 会话策略

### 短会话 vs 长会话

| | 短会话（<30 分钟） | 长会话（>1 小时） |
|---|---|---|
| 上下文 | 充裕 | 可能不足 |
| 质量 | 稳定 | 可能退化 |
| 适合 | 单一任务 | 大型重构 |
| 建议 | 默认选择 | 定期 /compact |

### 多会话并行

用多个终端窗口跑多个 Claude Code 会话：

```bash
# 终端 1
cd ~/Projects/my-app && claude  # 做功能开发

# 终端 2
cd ~/Projects/my-app && claude  # 做代码审查
```

两个会话完全独立，互不干扰。

### 会话命名（心理模型）

虽然 Claude Code 不支持给会话命名，但你可以在心里这样组织：

```
会话 A：「认证重构」—— 只做认证相关的改动
会话 B：「Bug 修复 #123」—— 只修这一个 bug
会话 C：「文档更新」—— 只更新文档
```

每个会话有清晰的职责，就像 Git 分支一样。
