# 第九章 性能调优与成本控制

## 响应速度优化

### 1. 选择合适的模型

| 任务 | 推荐模型 | 原因 |
|------|----------|------|
| 日常编码 | Sonnet 4.6 | 速度和质量平衡 |
| 复杂推理 | Opus 4.6 | 质量优先 |
| 简单搜索 | Haiku 4.5 | 速度优先 |
| 快速迭代 | Sonnet + /fast | 最快响应 |

### 2. 精简 CLAUDE.md

每轮对话都要处理 CLAUDE.md。从 5,000 tokens 精简到 2,500 tokens，每轮节省 2,500 tokens 的处理时间。

### 3. 定期 /compact

长对话累积的上下文拖慢每轮响应。20 轮后 `/compact` 一次。

### 4. 分段处理大文件

```
# 差：一次性读取整个大文件
读取 src/legacy/monolith.ts 并分析

# 好：指定需要的部分
读取 src/legacy/monolith.ts 的第 200-300 行，
分析 processPayment 函数
```

## 成本控制

### 查看用量

```
claude --usage
```

### 按任务控制

在对话中明确限制：
```
帮我写一个工具函数。简洁回答，不需要解释。
```

减少不必要的长回复。

### 模型降级策略

```
日常编码 → Sonnet（$3+$15/M）
简单问答 → /fast 模式
代码审查 → Sonnet
架构设计 → Opus（$5+$25/M）
```

80% 的任务用 Sonnet，20% 用 Opus，月度成本可控。

### 企业预算管理

| 指标 | 参考值 |
|------|--------|
| 单人日均 | ~$13 |
| 单人月均 | $150-250 |
| 90% 用户日均 | <$30 |

设置 API Key 的月度限额，防止意外失控。

## 常见性能问题

### "Claude 变慢了"

**原因**：上下文太长。
**解法**：`/compact` 或开新会话。

### "Claude 忘了之前说的"

**原因**：上下文被自动压缩。
**解法**：关键信息保存到文件而非只在对话中。

### "网络超时"

**原因**：网络不稳定或请求太大。
**解法**：检查网络，减少单次请求的文件量。
