# 第九章 性能调优与成本控制

## 影响响应速度的五个因素

```
响应时间 = 网络延迟 + 上下文处理 + 思考 + 生成 + 工具调用
```

每个因素都可以优化：

| 因素 | 影响 | 优化手段 |
|------|------|---------|
| 网络延迟 | 固定 ~100ms | 稳定网络、就近节点 |
| 上下文处理 | 与上下文长度成正比 | 精简 CLAUDE.md、/compact |
| 思考 | Extended Thinking 耗时 | 简单任务不需要 thinking |
| 生成 | 与输出长度成正比 | 要求简洁回答 |
| 工具调用 | 每次调用有开销 | 减少不必要的文件读取 |

## 响应速度优化

### 1. 选择合适的模型

| 任务 | 推荐模型 | 响应速度 | 质量 |
|------|----------|---------|------|
| 日常编码 | Sonnet 4.6 | ⚡⚡⚡ | ⭐⭐⭐⭐ |
| 复杂推理 | Opus 4.6 | ⚡⚡ | ⭐⭐⭐⭐⭐ |
| 简单搜索 | Haiku 4.5 | ⚡⚡⚡⚡ | ⭐⭐⭐ |
| 快速迭代 | Sonnet + /fast | ⚡⚡⚡⚡ | ⭐⭐⭐⭐ |

**经验法则**：先用 Sonnet，只在 Sonnet 质量不够时切 Opus。

### 2. 精简 CLAUDE.md

每轮对话都要处理 CLAUDE.md。数据对比：

| CLAUDE.md 大小 | 每轮额外开销 | 5 teammates 开销 |
|---------------|-------------|-----------------|
| 5,000 tokens | ~$0.025 | ~$0.125 |
| 2,500 tokens | ~$0.0125 | ~$0.0625 |
| 800 tokens | ~$0.004 | ~$0.02 |

精简到 800-2,500 tokens 是最佳区间。

### 3. 上下文管理

**定期 /compact**

```
# 20 轮对话后
/compact

# 带指示的压缩
/compact 保留关于认证模块重构的讨论，其他可以压缩
```

**何时开新会话**

| 信号 | 建议 |
|------|------|
| 对话超过 30 轮 | /compact 或开新会话 |
| 切换到完全不同的任务 | 开新会话 |
| Claude 开始"忘事" | /compact |
| 响应明显变慢 | /compact 或开新会话 |

**关键信息持久化**

```
# 差：关键信息只在对话中
"我们决定用 JWT 替代 Session，过期时间设为 24 小时"

# 好：写入文件
把这个决策记录到 docs/decisions/auth-jwt.md
```

写入文件后，即使开新会话也不会丢失。

### 4. 分段处理大文件

```
# 差：一次性读取整个大文件
读取 src/legacy/monolith.ts 并分析

# 好：指定需要的部分
读取 src/legacy/monolith.ts 的第 200-300 行，
分析 processPayment 函数
```

### 5. 减少不必要的工具调用

```
# 差：让 Claude 自己找
项目里有什么测试框架？

# 好：直接告诉它
项目用 vitest 做测试。帮我给 src/auth/verify.ts 写测试。
```

你已经知道的信息直接告诉 Claude，不要让它花时间读 package.json 去发现。

## 成本控制

### 定价速查

| 模型 | Input ($/M) | Output ($/M) | Thinking ($/M) |
|------|------------|-------------|----------------|
| Opus 4.6 | $5 | $25 | $25 |
| Sonnet 4.6 | $3 | $15 | $15 |
| Haiku 4.5 | ~$0.25 | ~$1.25 | - |

### 实际花费参考

| 场景 | 典型花费 |
|------|---------|
| 一次简单代码修改 | $0.05-0.20 |
| 一次代码审查 | $0.50-2.00 |
| 一次大型重构（1 小时） | $3-10 |
| 一天正常使用 | $5-30 |
| 一月（中度使用） | $150-250 |

### 七个省钱技巧

**1. 模型降级策略**

```
80% 日常任务 → Sonnet（$3+$15/M）
15% 复杂任务 → Opus（$5+$25/M）
5% 简单任务 → Haiku（$0.25+$1.25/M）
```

月度成本降低约 40%。

**2. /fast 模式**

```
/fast
```

同一个模型但更快输出。适合不需要深度思考的迭代。

**3. 简洁指令**

```
# 贵：Claude 会生成长回复
解释一下这段代码

# 便宜：Claude 只回答重点
这段代码有 bug 吗？只说有没有，如果有指出位置。
```

**4. 避免重复读文件**

```
# 贵：每轮都读
帮我改 auth.ts 的第 10 行（Claude 重新读文件）
再改第 20 行（Claude 又读一次）

# 便宜：一次说完
帮我改 auth.ts：
1. 第 10 行：把 timeout 从 5000 改为 10000
2. 第 20 行：添加错误日志
```

**5. Prompt Caching**

如果你通过 API 使用，确保利用 prompt caching（详见 Prompt 实战 Ch5）。缓存命中的 input 只收 10% 费用。

**6. 设置预算告警**

在 Anthropic Console 中设置：
- 月度预算上限
- 日用量告警阈值

**7. 分析用量模式**

```
claude --usage
```

查看哪些任务消耗最多，针对性优化。

### 企业预算管理

| 指标 | 参考值 |
|------|--------|
| 单人日均 | ~$13 |
| 单人月均 | $150-250 |


## 常见性能问题排查

### "Claude 变慢了"

**排查步骤**：
1. 对话轮数？→ >20 轮用 `/compact`
2. CLAUDE.md 多大？→ >3000 tokens 需精简
3. 是否在读大文件？→ 指定行号范围
4. 网络是否稳定？→ `ping api.anthropic.com`

### "Claude 忘了之前说的"

**原因**：上下文窗口接近上限，系统自动压缩了历史消息。

**解法**：
1. 关键决策写入文件（不只存在对话中）
2. 用 Memory 系统记住跨会话信息
3. 定期手动 `/compact`（比系统自动压缩更可控）

### "Claude 开始瞎改别的文件"

**原因**：对话太长，Claude 丢失了任务焦点。

**解法**：
1. 开新会话，重新描述任务
2. 在 prompt 中明确"只修改 src/auth/verify.ts，不要动其他文件"

### "同样的问题上次答得好这次答得差"

**原因**：上下文不同导致行为差异。

**解法**：
1. 重要的 prompt 保存为 Slash Command，确保每次一致
2. 关键上下文放在 CLAUDE.md 中，每次自动加载
