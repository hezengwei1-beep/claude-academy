# Recipes R06-R10：文档与分析

## R06: 多语言文档同步（完整版）

### 配置

需要 Filesystem MCP Server 访问 docs 目录：

```json
{
  "filesystem": {
    "command": "npx",
    "args": ["-y", "@modelcontextprotocol/server-filesystem", "./docs"]
  }
}
```

### 完整操作流程

```markdown
同步 docs/en/ 到 docs/zh/：

**Step 1: 扫描差异**
- 列出 docs/en/ 所有 .md 文件
- 列出 docs/zh/ 所有 .md 文件
- 找出三类：
  A. en 有 zh 没有（待翻译）
  B. en 修改时间 > zh 修改时间（待更新）
  C. 两边都有且 en 没更新（已同步）

**Step 2: 翻译新文件（A 类）**
- 逐个翻译
- 翻译规则：
  - 专业术语首次出现时：English (中文)，后续直接用缩写
  - 代码块不翻译
  - 保持 Markdown 格式完全一致
  - 链接中的路径不翻译，链接文字翻译

**Step 3: 更新过期文件（B 类）**
- 对比 en 和 zh 的 diff
- 只翻译变化的部分，不重新翻译整篇
- 保持已翻译部分的风格一致

**Step 4: 生成报告**
输出到 docs/sync-report.md：
| 文件 | 状态 | 操作 |
|------|------|------|
| getting-started.md | 新增 | 已翻译 |
| api-reference.md | 过期 | 已更新 Section 3 |
| faq.md | 同步 | 无需操作 |
```

## R07: API 设计审查 + OpenAPI 生成（完整版）

### Slash Command

```markdown
扫描 $ARGUMENTS 目录下的所有路由文件，生成 API 审查报告和 OpenAPI 规范。

**Phase 1: 发现所有端点**
- 扫描路由定义（Express/FastAPI/Next.js 格式）
- 提取：HTTP 方法、URL、参数、中间件

**Phase 2: RESTful 规范检查**
| 规则 | 检查项 |
|------|--------|
| URL 命名 | 复数名词？小写？连字符？ |
| HTTP 方法 | GET 只读？POST 创建？PUT 全量更新？PATCH 部分更新？DELETE 删除？ |
| 状态码 | 200/201/204/400/401/403/404/500 使用是否正确？ |
| 版本管理 | URL 中有 /v1/ 或 Header 中有版本？ |
| 分页 | 列表 API 有 limit/offset 或 cursor？ |
| 错误格式 | 错误响应是否统一格式？ |

**Phase 3: 一致性检查**
- 命名风格是否一致（camelCase vs snake_case）
- 认证方式是否统一
- 错误响应格式是否统一
- 分页格式是否统一

**Phase 4: 生成 OpenAPI 3.0**
输出到 docs/openapi.yaml，包含：
- 所有端点
- 请求/响应 Schema
- 认证方式
- 错误定义
```

## R08: 遗留代码现代化（完整版）

### 分析维度

```markdown
对 $ARGUMENTS 进行遗留代码分析：

**1. 语法层面**
- var → const/let
- 回调地狱 → async/await
- string 拼接 → template literal
- CommonJS require → ES import
- any 类型 → 具体类型

**2. 模式层面**
- God class（超大类）→ 拆分职责
- 深层嵌套 → 提前 return / guard clause
- 魔法数字 → 命名常量
- 重复代码 → 提取函数
- 手动状态管理 → 用 hooks 或状态库

**3. 依赖层面**
- 废弃的包 → 替代品
- 手写的工具函数 → 成熟的库（lodash/date-fns）
- 自制的 HTTP 客户端 → axios/fetch

**4. 输出格式**
每个改进点：
- 当前状态：代码片段
- 建议改为：代码片段
- 风险评估：低/中/高
- 优先级：P0（必须）/ P1（建议）/ P2（可选）
- 预估工作量：S/M/L

**5. 分阶段计划**
Phase 1（低风险，高收益）：语法现代化
Phase 2（中风险）：模式重构
Phase 3（需规划）：依赖替换
```

```markdown

**Step 1: 识别性能关键路径**
- 读取代码，找出被频繁调用的函数
- 识别 I/O 操作（数据库查询、文件读写、API 调用）
- 识别计算密集操作（循环、排序、加密）

**Step 2: 编写基准测试**

用 vitest bench：

```typescript
// benchmarks/auth.bench.ts
import { bench, describe } from 'vitest';

describe('认证性能', () => {
  bench('JWT 签名', () => {
    signToken({ userId: 1, role: 'user' });
  });

  bench('JWT 验证', () => {
    verifyToken(validToken);
  });

  bench('密码哈希', () => {
    hashPassword('test-password');
  });

  bench('密码验证', () => {
    comparePassword('test-password', hashedPassword);
  });
});
```

**Step 3: 运行并分析**
```bash
npx vitest bench
```

**Step 4: 输出报告**

| JWT 签名 | 50,000 | 0.02ms | 0.05ms | ✅ 好 |
| JWT 验证 | 100,000 | 0.01ms | 0.03ms | ✅ 好 |
| 密码哈希 | 10 | 100ms | 120ms | ⚠️ 正常（bcrypt 设计如此） |

**Step 5: 优化建议**

## R10: 安全扫描（完整版）

### 全面扫描清单

```markdown
执行全项目安全扫描：

**1. 依赖漏洞**
```bash
npm audit --json
```
分析每个漏洞的严重度和影响。

**2. 硬编码密钥搜索**
搜索以下模式：
- `password\s*=\s*['"]` （硬编码密码）
- `api[_-]?key\s*=\s*['"]` （硬编码 API Key）
- `secret\s*=\s*['"]` （硬编码 Secret）
- `-----BEGIN (RSA |EC )?PRIVATE KEY-----` （私钥文件）
- `ghp_[a-zA-Z0-9]{36}` （GitHub Token）
- `sk-[a-zA-Z0-9]{48}` （OpenAI/Anthropic Key）

**3. .gitignore 检查**
确认以下文件不在版本控制中：
- .env, .env.local, .env.production
- *.pem, *.key
- credentials.json
- node_modules/

**4. 代码安全检查**
- SQL 注入：搜索字符串拼接的 SQL
- XSS：搜索 dangerouslySetInnerHTML 和未转义的输出
- CSRF：检查表单是否有 CSRF token
- CORS：检查 CORS 配置是否过于宽松
- 路径遍历：检查文件操作是否验证路径

**5. 认证检查**
- JWT secret 是否足够强（>256 bit）
- Token 是否有过期时间
- 密码是否用 bcrypt/argon2 哈希
- 是否有 rate limiting

**6. 输出格式**
| 严重度 | 类型 | 位置 | 描述 | 修复方案 |
|--------|------|------|------|---------|
| 🔴 Critical | 硬编码密钥 | .env committed | .env 在 Git 中 | 移除 + 轮换密钥 |
| 🟡 High | SQL 注入 | src/db:42 | 字符串拼接 | 参数化查询 |
| ... | ... | ... | ... | ... |
```
