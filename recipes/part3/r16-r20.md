# Recipes R16-R20：大型工程与交付

## R16: Agent Teams 大型重构编排（完整版）

### 完整的操作手册

```markdown
进入 Delegate Mode。

**Phase 0: 分析（Lead 做）**
先分析 src/legacy/ 目录：
- 总文件数、总行数
- 依赖关系图
- 识别可以独立重构的模块

**Phase 1: 创建团队**
基于分析结果创建 4 人团队：

1. model-dev
   - 职责：数据模型层重构
   - 文件范围：src/legacy/models/ → src/models/
   - 技术：从 Sequelize 迁移到 Prisma
   - 完成标准：新 model 通过类型检查，旧 model 的所有字段都有对应

2. service-dev
   - 职责：业务逻辑层重构
   - 文件范围：src/legacy/services/ → src/services/
   - 依赖：model-dev 完成后开始
   - 完成标准：所有 service 方法迁移完成，接口签名文档化

3. api-dev
   - 职责：API 层重构
   - 文件范围：src/legacy/routes/ → src/api/
   - 依赖：service-dev 完成后开始
   - 完成标准：所有 endpoint 可访问，响应格式一致

4. test-dev
   - 职责：测试迁移
   - 文件范围：tests/legacy/ → tests/
   - 可并行：model 完成后立即开始写 model 测试
   - 完成标准：覆盖率 >= 原有水平

**Phase 2: 执行规则**
- 每人只改自己范围的文件
- 共享文件（types.ts, constants.ts）由 model-dev 统一管理
- 接口变更先发消息给下游 teammate
- 每完成一个模块，在任务列表标记
- 每 30 分钟 Lead 检查一次进度

**Phase 3: 验证（Lead 做）**
所有 teammate 完成后：
1. 运行全量测试
2. 检查类型检查通过
3. 检查 lint 通过
4. 对比新旧代码的接口一致性
5. 生成重构报告
```

### 预期时间线

```
0:00  Lead 分析代码结构
0:10  创建团队 + 分配任务
0:15  model-dev 开始 | test-dev 写 model 测试
0:35  model-dev 完成 → service-dev 开始
0:55  service-dev 完成 → api-dev 开始
1:15  api-dev 完成 | test-dev 完成集成测试
1:25  Lead 验证 + 报告
~1:30 完成
```

单 agent 预估：4-6 小时。Agent Teams：~1.5 小时。

```
3. 如果有新部署，分析改动是否可能导致性能问题
4. 给出止血方案（不要自动执行，只给方案）
5. 评估是否需要回滚
```

### 预期响应（你在手机上看到）

```
分析完成：

**发现**：30 分钟前有一次部署（commit abc123）
改动内容：src/db/queries/orders.ts 中新增了一个未加索引的 JOIN

**止血方案**：
方案 A（推荐）：回滚到上一个版本
  命令：git revert abc123 && npm run deploy
  风险：低，新功能暂时不可用
  预计恢复时间：5 分钟

方案 B：紧急加索引
  SQL：CREATE INDEX CONCURRENTLY idx_orders_user_created ON orders(user_id, created_at);
  风险：中，CONCURRENTLY 不锁表但需要时间
  预计恢复时间：10-15 分钟

建议选方案 A 先止血，然后方案 B 作为正式修复。
```

## R18: 完整 Feature 交付流水线（完整版）

### 端到端流程

```markdown
功能：$ARGUMENTS

**Stage 1: 需求分析（/plan 模式）**
1. 解析功能描述
2. 拆解为技术任务
3. 识别涉及的模块和文件
4. 评估复杂度和工作量
5. 输出任务清单，等我确认

**Stage 2: 技术设计（确认后）**
1. 数据模型设计（如果需要）
2. API 接口设计（请求/响应格式）
3. 组件/模块划分
4. 错误处理策略
5. 输出设计文档，等我确认

**Stage 3: 实现（确认后，用 Agent Teams）**
进入 Delegate Mode，创建团队：
- 按设计的模块划分 teammates
- 设置文件边界和依赖
- 并行开发

**Stage 4: 测试**
- 单元测试（各 teammate 自己写）
- 集成测试（test-dev teammate）
- 运行全量测试确认无回归

**Stage 5: 文档**
- 更新 API 文档
- 更新 README（如果需要）
- 生成变更日志

**Stage 6: 交付准备**
- 生成 PR 描述（标题 + 摘要 + 测试计划）
- 代码自审（运行 /review）
- 输出最终确认清单
```

## R19: 代码库全景搜索（完整版）

```markdown
深度搜索"$ARGUMENTS"相关的所有代码：

**Level 1: 直接匹配**
- 文件名包含关键词的文件
- 代码中包含关键词的位置（文件:行号）

**Level 2: 语义关联**
- 函数/类的命名虽不含关键词，但功能相关
- 配置文件中的相关设置
- 环境变量中的相关项

**Level 3: 调用链**
- 入口点在哪里？
- 调用链：谁调用了谁？
- 数据流：数据从哪来到哪去？

**Level 4: 依赖**
- 使用了哪些外部包？
- 配置了哪些第三方服务？
```
