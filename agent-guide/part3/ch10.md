
# 第十三章 Plan → Team 工作流

## 先想清楚，再动手

多代理团队最大的浪费不是 token 费用，而是**方向错误后的返工**。5 个 teammates 同时朝错误的方向冲刺，造成的浪费是单 agent 的 5 倍。

Plan → Team 工作流的核心理念：**先用一个 agent 想清楚怎么做，再用团队并行执行。**

## 工作流程

```
第一阶段：Plan（规划）
┌────────────────────────────┐
│  你 → Lead：描述任务        │
│  Lead：分析代码库          │
│  Lead：制定执行计划        │
│  你：审查并批准计划        │
└──────────┬─────────────────┘
           │
第二阶段：Team（执行）
┌──────────▼─────────────────┐
│  Lead：spawn teammates     │
│  Lead：分配任务            │
│  Teammates：并行执行       │
│  Lead：协调和综合          │
└────────────────────────────┘
```

## 第一阶段：Plan

### 使用 /plan 命令

```
/plan

我需要将项目的认证系统从 Session-based 迁移到 JWT。
涉及的代码在 src/auth/, src/middleware/, src/routes/。
测试在 tests/auth/。

请分析当前代码，制定迁移计划，包括：
1. 需要修改哪些文件
2. 修改的先后顺序（哪些可以并行）
3. 风险点和注意事项
4. 建议的团队分工
```

### Lead 的规划产出

一个好的计划应该包含：

```markdown
## 迁移计划：Session → JWT

### 1. 文件变更清单
- src/auth/session.ts → 删除
- src/auth/jwt.ts → 新建
- src/auth/middleware.ts → 重写
- src/middleware/auth.ts → 修���
- src/routes/auth.ts → 修改
- config/auth.ts → 修改
- tests/auth/*.test.ts → 全部重写

### 2. 执行顺序
Phase 1（可并行）：
- jwt.ts 核心实现
- config/auth.ts 配置更新

Phase 2（依赖 Phase 1）：
- middleware.ts 重写
- routes/auth.ts 修改

Phase 3（依赖 Phase 2）：
- 测试更新
- 集成测试

### 3. 风险点
- 数据库中现有的 session 记录需要处理
- 已登录用户会在部署时被强制登出
- JWT secret 的安全存储

### 4. 建议团队分工
- core-dev：Phase 1 的 jwt.ts 和 config
- middleware-dev���Phase 2 的中间件和路由
- test-dev：Phase 3 的测试（可以从 Phase 1 开始写单元测试）
```

### 你的审查

这是你介入的关键时刻。检查：

- [ ] 文件清单是否完整？有没有遗漏的？
- [ ] 执行顺序是否合理？依赖关系对吗？
- [ ] 风险点你都能接受吗？
- [ ] 团队分工的文件边界清晰吗？

确认后，进入第二阶段。

## 第二阶段：Team

### 带着计划创建团队

```
计划批准。请按计划执行：

1. 进入 Delegate Mode
2. 创建以下团队：
   - core-dev：负责 src/auth/jwt.ts 和 config/auth.ts
   - middleware-dev：负责 src/auth/middleware.ts 和 src/routes/auth.ts
   - test-dev：负责 tests/auth/ 目录

3. 任务依赖：
   - middleware-dev 的工作依赖 core-dev 完成
   - test-dev 可以先写 jwt.ts 的单元测试（与 Phase 1 并行），
     集成测试等 Phase 2 完成后再写

4. 规则：
   - 每个 teammate 修改文件前先发消息确认接口定义
   - 每完成一个 phase 在任务列表标记
   - 遇到计划中没覆盖的情况，先发消息给 Lead 讨论
```

### 让 teammates 先提交执行计划

更谨慎的做��——要求每个 teammate 在动手前先说明自己打算怎么做：

```
在开始实际修改之前，每个 teammate 先提交自己的执行计划：
- 打算修改哪些文件的哪些部分
- 新增/删除/修改的函数和接口
- 预计的测试策略

所有 teammate 提交计划后，我来审批。通过后才能动��。
```

这增加了一轮沟通开销，但大大减少了返工风险。适合：
- 涉及核心模块的改动
- 不可逆的变更（删除、重命名）
- 多人之间有接口依赖的场景

## Token 成本分析

Plan → Team 工作流的成本结构：

| 阶�� | 成本 |
|------|------|
| Plan（单 agent 规划） | 1x |
| Team（3 teammates 执行） | 3x |
| 加上 Delegate Mode Lead | +0.5x |
| 如果要 teammates 提交计划 | +1x |
| **总计** | 约 4.5-5.5x |

对比不做规划直接上团队：3-4x token 但有更高的返工风险。

**经验法则**：如果任务复杂度值得用 Agent Teams，那一定值得花额外 30% 的成本做规划。

## Plan → Team 的最佳场景

| 场景 | 适合 Plan → Team 吗？ |
|------|----------------------|
| 简单的并行代码审查 | ❌ 不需要，直接上 Team |
| 大型模块重构 | ✅ 强烈推荐 |
| 新功能开发（涉及多模块） | ✅ 推荐 |
| Bug 修复（单文件） | ❌ 用单 agent |
| 技术栈迁移 | ✅ 强烈��荐 |
| 性能优化（需要先分析后实施） | ✅ 推荐 |

## 常见陷阱

### 陷阱一：规划过度

规划的目的是减少方向性错误，不是消除所有不确定性。如果你发现规划阶段已经花了半小时还在细化，可能是过度了。

**经验**：好的计划在 5-10 分钟内产出。如果更久，说明任务可能需要先拆分。

### 陷阱二：计划不更新

执行过程中一定会发现计划没有覆盖的情况。这时候需要**暂停执行、更新计划、再继续**，而不是让 teammates 各自临场发挥。

### 陷阱三：审批环节变成瓶颈

如果你要求每个 teammate 都先提交计划等审批，而你没有及时审批，整个团队就会空转。

**解决**：只在高风险的关键步骤设置审批点，不是每一步都要审批。
