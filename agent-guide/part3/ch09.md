# 第十二章 Delegate Mode：纯协调者模式

## 什么是 Delegate Mode

Delegate Mode 是 Agent Teams 中最强大也最被低估的功能。

在默认模式下，Team Lead 既可以自己干活，也可以协调 teammates。这听起来很灵活，但实际上是一个陷阱——当 Lead 自己开始写代码，它的上下文窗口会被代码占满，协调能力急剧下降。

Delegate Mode（通过 `Shift+Tab` 切换）让 Lead **完全退出执行**，只做协调：

| | 默认模式 | Delegate Mode |
|---|---|---|
| Lead 写代码 | ✅ 可以 | ❌ 不可以 |
| Lead 读文件 | ✅ 可以 | ❌ 不可以 |
| Lead spawn teammates | ✅ ���以 | ✅ 可�� |
| Lead 发消息 | ✅ 可以 | ✅ 可以 |
| Lead 管理任务 | ✅ 可以 | ✅ 可以 |
| Lead 关闭 teammates | ✅ 可以 | ✅ 可以 |

## 为什��� Delegate Mode 更好

### 原因一：上下文保护

Lead 的上下文窗口是整个团队的"大脑"。如果 Lead 自己读了 10 个文件、写了 500 行代码，它的上下文就被占满了。当 teammates 发来消息或汇报发现时，Lead 可能已经没有足够的上下文来理解和综合��些信息。

Delegate Mode 保证 Lead 的上下文始终留给**协调**——理解 teammates 的汇报、做决策、处理交叉发现��

### 原因二：避免角色混乱

当 Lead 同时做协调和执行时，容易发生：
- Lead 在写代码时忽略了 teammate 发来的消息
- Lead 修改的文件与 teammate 正在修改的文件冲突
- Lead 的注意力在"自己的代码"和"团队管理"之间分裂

Delegate Mode 消除了这个问题——Lead 只有一个职责：管理团队。

### 原因三：这是社区验证的最佳实践

多位社区专家独立得出了同一个结论：

> "Delegate mode + 明确文件边界 = 消除最常见的 Agent Teams 失败模式"

这两个实践组合在一起，可以避免绝大多数新手在使用 Agent Teams 时遇到的问题。

## 如何使用 Delegate Mode

### 进入 Delegate Mode

在 Claude Code 会话中按 `Shift+Tab`，或在对话中明确说：

```
进入 Delegate Mode。从现在开始，你只做协调，不要自己读文件或写代码。
所有执行工作都交给 teammates。
```

### Delegate Mode 下的工作流

```
1. 你描述任务给 Lead
   ↓
2. Lead 分析任务，制定团队计划
   ��
3. Lead spawn teammates，分配职责和文件边界
   ↓
4. Lead 创建任务列表，设置依赖
   ↓
5. Teammates 开始工作
   ↓
6. Lead 监控进度，转发跨 teammate 的信息
   ↓
7. Lead 处理 teammates 的汇报和��题
   ↓
8. Lead 综合所有结果，向你报告
```

### 实战示例：Delegate Mode 协调 5 人重构

```
进入 Delegate Mode。

我需要重构用户管理模块。当前代码在 src/users/ 下，
有 controller、service、repository、model、validation 五层。

请创建一个 5 人团队：

1. model-dev
   - 负责：src/users/models/
   - 任务：将 User model 从 Sequelize ORM 迁移到 Prisma
   
2. repo-dev
   - 负��：src/users/repositories/
   - 任务：重写 repository 层适配新的 Prisma model
   - 依赖：model-dev 完成后开始

3. service-dev
   - 负责：src/users/services/
   - 任务：更新 service 层，使用新的 repository 接口
   - 依赖：repo-dev 完成后开始

4. controller-dev
   - 负责：src/users/controllers/ 和 src/users/routes/
   - 任务：确保 controller 层与新 service 接口兼容
   - 依赖：service-dev 完成后开始

5. test-dev
   - 负责：tests/users/
   - 任务：更新所有测试适配新的实现
   - 可以与其他开发者并行，每完成一层就更新对应的测试

执行规则：
- 每个人只改自己负责的文件
- 修改接口时先发消息给依赖方确认
- 每完成一个子任务在任务列表标记
- 遇到设计决策先发消息给我，不要自作主张
```

在这个例子中，Lead 全程不碰代码。它的角色是：
- 监控五个 teammates 的进度
- 当 model-dev 修改了接口定义，确保 repo-dev 知道
- 当出现设计争议，做裁���
- 当一切完成，综合检查整体一致性

## 何时不用 Delegate Mode

Delegate Mode 不是银弹。以下场景用默认��式更好：

1. **团队只有 1-2 人**：协调开销大于收益，Lead 不如自己也干活
2. **任务非常简单**：不需要复杂协调的快速任务
3. **需要 Lead 亲自验证**：Lead 需要读代码来判断 teammates 的产出质量

## 经验法则

| 团队规模 | 建议模式 |
|----------|----------|
| 1-2 teammates | 默认模式（Lead 也干活） |
| 3-5 teammates | Delegate Mode（Lead 纯协调） |
| 5+ teammates | 强烈建议 Delegate Mode |

## Delegate Mode 的心智模型

把自己想象成**项目经理**，不是**技术经理**：

- 项目经理不写代码，但确保每个人都在正确的方向上工作
- 项目经理处理的是"谁应该做什么"、"这两个人的工作怎么协调"、"这个决策影响了谁"
- 项目经理的价值在于全局视野和协调能力，不在于写更多代码
