## 创建团队的自然语言指令

Agent Teams 的交互完全基于自然语言。你不需要记住 API 或命令语法——直接用人话告诉 Claude 你想要��么。

### 基础模板

```
创建一个 [N] 人团队来 [任务目标]：
- [teammate-名称]：[职责描述]
- [teammate-名称]：[职责描述]
- [teammate-��称]：[职责描述]

[额外约束和期望]
```

### 好的 spawn 指令 vs 坏的

**差**：
```
帮我审查代码，搞几个 agent。
```
问题：没有明确角色、没有职责划分、没有文件边界。

**好**：
```
创建一个 3 人代码审查团队来审查 src/auth/ 模块的改动：
- security-reviewer：专注安全漏洞，检查 src/auth/middleware.ts 和 src/auth/token.ts
- logic-reviewer：检查业务逻辑正确性，重点看 src/auth/handlers/ 目��
- test-reviewer：验证测试覆盖率，检查 tests/auth/ 目录

要求：
1. 每人在自己的范围内工作，不要越��
2. 发现问题时按严重度标记（Critical/High/Medium/Low）
3. 完成后在任务列表标记完成，并给我发消息汇总
```

### 关键要素清单

一个好的团队创建指令应该包含：

- [ ] **团队规模**：3-5 人（甜区）
- [ ] **每人的名称**：简短有意义
- [ ] **每人的职责**：一句话说清楚做什么
- [ ] **文件边界**：明确每人负责的文件/目录
- [ ] **产出格式**：期望什么样的输出
- [ ] **完成标准**：什么算"做完了"

## 任务管理

### 创建任务

Team Lead 或 teammates 都可以创建任务：

```
添加以下任务到任务列表：
1. 审查 JWT token 的生成逻辑 [分配给 security-reviewer]
2. 基准测试认证接口的响���时间 [分配给 perf-reviewer]
3. 检查 token 过期的边界测�� [分配给 test-reviewer]

任务 2 和 3 依赖任务 1 完成后才能开始。
```

### 查看任务状态

按 `Ctrl+T` 或输入：
```
显示当前任务列表状态
```

### 任务生命周期

```
pending（待处理）
  │
  ▼ teammate 认领
in_progress（进���中）
  │
  ▼ teammate 完成
completed（已完成）
```

### 任务依赖

依赖的任务会自动阻塞，直到前置任务完成：

```
任务 A: 设计 API schema → pending → in_progress → completed ✓
                                                      │
任务 B: 实现 API endpoints → blocked → pending → in_progress → ...
```

## 队友通信

### Lead → Teammate

```
给 security-reviewer 发消息：请重点关注 OAuth 回调 URL 的验证逻辑
```

### Teammate → Lead

Teammates 完成任务后通常会主动向 Lead 报告：
```
[security-reviewer → Lead]：审查完成，发现 2 个 Critical 问题：
1. OAuth redirect_uri 没有做白名单校验
2. Refresh token 没有绑定客户端 IP
```

### Teammate → Teammate

```
[security-reviewer → perf-reviewer]：
我发现 token 验证每次都查数据库，没有缓存。
这可能就是你看到的性能瓶颈的原因，建议你看看 src/auth/verify.ts:42
```

这种点对点的交叉发现是 Agent Teams 最有价值的能力之一。

## 快捷键完整操作

| 快捷键 | 作用 | 使用场景 |
|--------|------|----------|
| `Shift+Down` | 切换到下一个 teammate | 浏览不同 teammate 的工作 |
| `Shift+Up` | 切换到上一个 teammate | 返回查看之���的 teammate |

## 实战示例：3 人并行代码审查

以下是一个完整的实操流程。

### 第一步：创建团队

```
我要审查 PR #287，这个 PR 修改了用户认证模块。
请创建一个 3 人审查团队：

1. security-auditor
   - 职责：检查安全漏洞
   - 关注文件：src/auth/*.ts, src/middleware/auth.ts
   - 重点：SQL 注入、XSS、认证绕过、敏感信息泄露

2. perf-analyst
   - 职责：分析性能影响
   - ��注文件：src/auth/*.ts, src/db/queries/auth.ts
   - 重点：N+1 查询、缺失索引、不必要的数据库调用

3. test-checker
   - 职责：验证测试质量
   - 关注文件：tests/auth/**/*.test.ts
   - 重点：覆盖率、边界情况、mock 是否合理

每人完成后：
- 在任务列表标记完成
- 给我发消息，格式为：[角色] 发现 X 个问题（N Critical / M High / ...）
- 如果发现与其他人职责相关的问题，直接发消息给对方
```

### 第二步：监控进度

使��� `Ctrl+T` 查看任务列表：

```
[ ] 安全审查 - security-auditor - in_progress
[ ] ���能分析 - perf-analyst - in_progress
[ ] 测试检查 - test-checker - in_progress
```

使用 `Shift+Down/Up` 轮流查看每个 teammate 的工作进展���

### 第三步：处理交叉发现

security-auditor 发现认证中间件对每个请求都做全量 token 验证（安全角度无问题，但有性能隐患）。它直接给 perf-analyst 发消息：

```
[security-auditor �� perf-analyst]：
src/middleware/auth.ts:67 的 verifyToken() 每次都查 DB，
从安全角度这是对的，但你可能要评估性能影响。
考虑是否可以加 token 缓存 + 短过期时间。
```

### 第四步：汇总结果

三人都完成后，你会收到各自的汇总。然后可以让 Lead 综合：

```
请综合三位审查者的发现，生成最终的审查报告：
- 按严重度排序
- 标注哪些发现是交叉相关的
- 给出修复优先级建议
```

### 关键收获

这个案例展示了 Agent Teams 的三个核心价值：

1. **并行**：三个维度同时审查，总耗时约等于最慢的那个
2. **专业化**：每个 teammate 只关注一个维度，更深入
3. **交叉发现**：teammates 之间直接通信，发现单人难以察觉的关联问题
