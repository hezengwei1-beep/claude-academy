
# 第十四��� 文件边界与冲突管理

> 社区共识：Agent Teams 最常见的失败原因就是两个 teammates 同时修改同一个文件。

本章解决这个问题。

## 核心原则

**一个文件同一时间只能有一个 teammate 修改。**

这不是技术限制，而是实操原则。技术上两个 teammate 可以同时修改同一个文件，但后果通常是灾难性的：

- Git merge 冲突
- 互相覆盖对方的改动
- 修改逻辑上互相矛盾
- 调试时难以定位哪个 teammate 引入了问题

## 在 spawn 时指定文件范围

最有效的预防手段：**创建 teammate 时就明确它的文件管辖范围**。

### 目录级隔离

最简单的方式——每个 teammate 负责一个独立目录：

```
创建团队：
- frontend-dev：只修改 src/components/ 和 src/pages/ 下的文件
- backend-dev：只修改 src/api/ 和 src/services/ 下的文件
- db-dev：只修改 src/db/ 和 migrations/ 下的文件

规则：不要修改自己范围外的文件。如果需要修改，先发消息给对应的 teammate 协商。
```

### 文件级隔离

当多人需要修改同一目录时，细化到文件级别：

```
以下人员都在 src/auth/ 目录工作，但分工不同：
- token-dev：只修改 token.ts, token.test.ts
- session-dev：只修改 session.ts, session.test.ts
- middleware-dev：只修改 middleware.ts, middleware.test.ts

共享文件 types.ts 和 index.ts 由 middleware-dev 统一修改。
其他人需要改类型定义时，发消息给 middleware-dev 代为修改。
```

### 接口边界约定

不同 teammates 负责的模块之间通过接口连接。修改接口时必须双方确认：

```
接口约定规则：
- 修改导出的函数签名前，先发消息给所有使用方
- 新增接口可以自行决定，但要通知相关方
- 删除接口必须确认没有调用方后才能操作
```

## Worktree 隔离模式

对于特别重要或风险高的场景，可以让每个 teammate 在独立的 Git worktree ��工作：

```
创建团队，每个 teammate 使用独立的 worktree：
- feature-dev：在 worktree feature-branch 中工作
- refactor-dev：在 worktree refactor-branch 中工作
- test-dev：在 worktree test-branch 中工作

完成后各自提交到自己的分支，最后由 Lead 协调合并。
```

**优点**：
- 完全消除文件冲突（不同的物理文件副本）
- 每个 teammate 可以自由修改任何文件
- Git 历史清晰，每个分支对应一个 teammate 的工作

**缺点**：
- 合并成本更高（最终需要 merge 多个分��）
- 无法实时共享中间成果
- 适合独立性强的任务，不适合需要频繁协调的任务

## Git 冲突的处理策略

即使设置了文件边界，偶尔还��会出现冲突。处理策略：

### 预防：共享文件指定唯一修改者

```
以下共享文件指定唯一修改者：
- package.json → infra-dev
- tsconfig.json → infra-dev
- src/types/index.ts → types-dev
- .env.example → infra-dev

其他人如需修改这些文件，提出请求，由指定修改者统一处理。
```

### 处理：让 agent 自行解决

如果冲突已经发生，可以指示 teammate：

```
你的改动与 [其他 teammate] 冲突了。
请 pull 最新代码，保留双方的改动（不要丢弃任何一方的工作），
解决冲突后重新提交。
如果不确定如何保留对方的意图，发消息给对方确认。
```

### 升级：Lead 介入

复杂冲突需要 Lead 做裁决：

```
[teammate-A] 和 [teammate-B] 在 src/auth/middleware.ts 上有冲突。
A 把认证改成了 async 模式，B 增加了 rate limiting。
两个改动都需要保留。

Lead 请协调：让 A 先提交 async 改动，然后 B 在此基础上加 rate limiting。
```

## 反面案例：不设边界的后果

以下是一个真实的社区案例（简化版）：

### 场景

3 个 teammates 被要求"重构用户模块"，没有指定文件边界。

### 发生了什么

1. Teammate A 重命名了 `getUserById` → `findUser`
2. Teammate B 在 `getUserById` 的基础上添加了缓存��辑
3. Teammate C 修改了调用 `getUserById` 的测试文件

三个 teammate 同时提交后：
- A 的代码中 `getUserById` 不存在了
- B 的缓存代码引用了不存在的函数名
- C 的测试调用了不存在的函数名
- 整个模块编译失败

### 修复成本

- 排查冲突：20 分钟
- 手动合并三方改动：40 分钟
- 重新运行测试：10 分钟
- 总计浪费约 1 小时 + 大量 token

### 如果设了边界

```
- A：负责 src/users/models/ → 重命名 model 层
- B：负责 src/users/services/ → 添加缓存逻辑
- C：负责 tests/users/ → 更新测试
- 约定：A 改完接口后通知 B 和 C 新的函数名
```

按顺序执行，零冲突。

## 最佳实践清单

1. **创建 teammate 时必须指定文件范围**——这是第一优先级
2. **共享文件指定唯一修改者**——package.json、类型定义、配置文件
3. **接口变更需要通知**——修改导出的函数签名前先沟���
4. **高风险场景用 Worktree 隔离**——大型重构、技术迁移
5. **设置清晰的提交顺序**——有依赖的改动按顺序��交
6. **Delegate Mode + 文件边界 = 最佳组合**——Lead 纯协调 + 每人有明确的文���范围

## 文件边界的思维模型

把每个 teammate 想象成一个独立的微服务：
- 它有自己的"数据库"（负责的文件）
- 它通过"API"与其他服务通信（Mailbox）
- 它不应该直接访问其他服务的"数据库"（不修改别人的文件）

这个思维模型在绝大多数场景下都能帮你做出正确的决策。
