# 第十六章 Git Worktrees 并行开发

## 什么是 Git Worktrees

Git Worktree 是 Git 原生功能（不是 Claude 发明的），允许你在同一个仓库中同时维护多个工作目录，每个目录对应一个不同的分支。

**类比**：如果把 Git 仓���比作一本书，普通的 checkout 是"一次只能翻到一页"。Worktree 让你同时打开多个书签，每个书签在不同的页面上。

## 在 Claude Code 中使用 Worktrees

### 快捷方式

```bash
# 指定名称创建 worktree
claude --worktree feature-auth

# 指定名称，用于 bug 修复
claude --worktree bugfix-123

# 自动生成名称
claude --worktree
```

Claude Code 会自动：
1. 创建一个新的 Git worktree
2. 在新的 worktree 中启动一个 Claude Code 会话
3. 完成后自动清理（如果没有做任何改动）

### 手动方式

```bash
# 手动创建 worktree
git worktree add ../my-project-feature-a -b feature-a

# 进入 worktree 并启动 Claude Code
cd ../my-project-feature-a
claude
```

## 核心特点

### 完全独立

每个 worktree 是一个独立的代码副本：
- 独立的分支
- 独立的文件系统
- 独立的 Claude Code 会话
- 独立的上下文窗口

在 worktree A 中的任何操作都不会影响 worktree B。

### 共享 Git 历史

虽然工作目录独立，但所有 worktree 共享同一个 Git 仓库：
- `.git` 对象数据库是��享的
- 可以看到其他分支的提交历史
- 可以 merge 或 rebase 其他分支的改动

### 自动清理

如果一个 worktree 会话结束时没有任何文件改动，Claude Code 会自动删除这个 worktree。有改动的 worktree 会保留。

## .worktreeinclude 文件

默认情况下，新的 worktree 不包含 `.gitignore` 忽略的文件（如 `node_modules/`、`.env`、构建产物等）。这意味着新 worktree 可能无法直接运行——缺少依赖。

**解决方案**：在项目根目录创建 `.worktreeinclude` 文件，指定需要复制到新 worktree 的文件：

```
# .worktreeinclude
node_modules/
.env
.env.local
dist/
```

## 实战示例

### 场景一：新功能 + 紧急修复并行

你正在开发一个新功能，突然来了一个线上 bug 需要紧急修复。

**不用 worktree**：
```
git stash        # 保存当前工作
git checkout main
git checkout -b hotfix-123
# 修复 bug...
git checkout feature-branch
git stash pop    # 恢复工作
```

这个流程有风险：stash 可能冲突，context switch 浪费时间。

**用 worktree**：
```bash
# 终端 1：继续新功能开发（不受影响）
# 当前目录就是你的功能分支

# 终端 2：开一个新的 worktree 修复 bug
claude --worktree hotfix-123
```

两个 Claude Code 会话完全独立，互不干扰。新功能的开发进度不受影响。

### 场景二：探索多种技术方案

不确定用 Redis 还是 Memcached 做缓存？

```bash
# 方案 A
claude --worktree explore-redis
# "请在这个分支中用 Redis 实现缓存层"

# 方案 B
claude --worktree explore-memcached
# "请在这个分支中用 Memcached 实现缓存层"
```

两个方案并行实现，比较后选择更好的。

### 场景三：长期重构 + 日常开发

大型重构可能持续数天。用 worktree 隔离，日常开发不受影响：

```bash
# 重构分支（可以跑几天）
claude --worktree refactor-v2

# 日常开发（正常节奏）
claude  # 在主分支上
```

## Worktrees vs Agent Teams

| 维度 | Worktrees | Agent Teams |
|------|-----------|-------------|
| 隔离级别 | 完全隔离（独立文件副本） | 逻辑隔离（共享文件系统） |
| 通信方式 | 无（手动 merge） | Mailbox 直接通信 |
| 协调方式 | 手动（你在终端间切换） | 自动（Lead 协调） |
| 适合场景 | 完全独立的任务 | 需要协调的协作任务 |
| 成本 | 每个 worktree = 1x | 团队 = 3-7x（因为有协调开销） |
| 合并成本 | 较高（事后 merge） | 较低（实时协调） |

**经验法则**：
- 两个任务**完全独立**（不同模块、不同分支）→ Worktrees
- 两个任务**需要协调**（共享模块、相互依赖）→ Agent Teams

## 注意事项

1. **不要在同一分支创建多个 worktree**：Git 不允许
2. **记得安装依赖**：新 worktree 可能需要 `npm install`
3. **环境变量**：检查 `.env` 文件是否需要复制
4. **构建产物**：可能需要重新构建

Worktrees 是最"轻量"的并行方案——没有额外的配置成本，没有实验性功能的不确定性，开箱即用。
