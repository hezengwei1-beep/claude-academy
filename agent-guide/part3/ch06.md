# 第九�� 架构���析：四组件模型

Agent Teams 不是一个简单的"多开几个 Claude"。它是一个有明确架构设计的协作系统，由四个核心组件构成。

## 组件一：Team Lead（团队负责人）

Team Lead 就是你启动的主 Claude Code 会话。它的职责：

- **创建团队**：spawn teammates，指定每个 teammate 的职责
- **分配任务**：将任务添加到共享任务列表
- **协调工作**：监控进度，解决冲突，做方向决策
- **综合结果**：汇总各 teammate 的产出，形成最终结果

**类比**：如果把 Agent Teams 比作一个软件项目组，Team Lead 就是技术经理（Tech Lead）——它可以自己写代码，也可以纯做协调。后面我们会看到，纯协调的 Delegate Mode 通常是更好的选择。

**代码示例——创建团队**：
```
创建一个 3 人团队来审查 PR #142：
- security-reviewer：专注安全漏洞和认证问题
- perf-reviewer：检查性能影响和资源使用
- test-reviewer：验证测试覆盖率和边界情况

每人审查完成后，在任务列表中标记完成，并发消息给我汇总发现。
```

## 组件二：Teammates（队友）

每个 Teammate 是一个**独立的 Claude Code 实例**，具有以下特性：

### 独立的上下文窗口

每个 teammate 有自己的上下文窗口，彼此隔离。这意味着：
- teammate A 读的文件不会出现在 teammate B 的上下文中
- 每个 teammate 可以专注于自己负责的代码区域
- 上下文窗口利用率更高（从单 agent 的 80-90% 降到 ~40%）

### 独立的工具权限

每个 teammate 可以：
- 读写文件
- 运行命令
- 搜索代码
- 使用 subagents（但不能创建子团队）

### 生命周期

```
spawn（创建） → 工作 → 完成/被关闭
```

teammate 一旦被关闭，其上下文即丢失。当前版本不支持恢复已关闭的 teammate。

## 组件三：Task List（共享任务列表）

Task List 是整个 Agent Teams 的**协调中枢**。所有 teammates 共享同一个任务列表。

### 任务结构

每个任务包含：
- **标题**：简要描述
- **描述**：详细要求
- **状态**：pending → in_progress → completed
- **负责人**：哪个 teammate 认领了这个任务
- **依赖**：这个任务依赖哪些其他任务先完成

### 任务依赖与自动解锁

你可以设置任务之���的依赖关系：

```
任务 1: 设计数据库 schema（无依赖）
任务 2: 实现 API endpoints（依赖任务 1）
任务 3: 编写前端组件���依赖任务 2）
任务 4: 编写集成测试（依赖任务 2 和任务 3）
```

当任务 1 完成后，任务 2 自动解锁，相关 teammate 可以开始工作。

### 自行认领 vs 指定分配

两种模式都支持：

- **指定分配**：Team Lead 明确指定"任务 1 给 teammate-A"
- **自行认领**：Team Lead 只创建任务，teammates 自行认领未被认领的任务

社区实践表明，**指定分配 + 明确文件边界**的效果优于自行认领。

## 组件四：Mailbox（邮箱）

Mailbox 是 teammates 之间**直接通信**的机制。这是 Agent Teams 区别于 Subagents 的核心特性。

### 通信方式

```
# Lead 给 teammate 发消息
给 security-reviewer 发消息：请重点检查 JWT token 的过期处理逻辑

# Teammate 给 Lead 回报
给 lead 发消息：发现 auth middleware 中 token 刷新有竞态条件

# Teammate 之间通信
给 perf-reviewer 发消息：我发现的这个安全问题可能和你看到的性能问题有关，
auth 模块在每次请求时都重新验证 token，建议你一起看看
```

### 为什么需要 Mailbox

在真实的协作中，发现往往是交叉的：

- 安全审查者发现一个认证问题，这恰好解释了性能审查者发现的慢查询
- 前端开发者需要后端开发者先暴露一个 API
- 测试者发现的 bug 可能影响多个模块的开发者

没有 Mailbox，这些交叉发现只能通过 Lead 中转——效率低且信息可能失真。有了 Mailbox，teammates 可以直接点对点沟通。

## 与微服务架构的类比

如果你有软件架构背景，可以这样理解 Agent Teams：

| Agent Teams 概念 | 微服务类比 |
|---|---|
| Team Lead | API Gateway / Orchestrator |
| Teammates | 微服务实例 |
| Task List | 消息队列（RabbitMQ/Kafka��� |
| Mailbox | 服务间 RPC 调用 |
| 独立上下文 | 服务隔离（每个服务有自己的数据库） |

两者的核心理念相同：**通过隔离和通信来管理复杂性**。

## 架构的局限性

了解局限和了解能力同样重要：

1. **不支持嵌套团队**：teammate 不能创建自己的子团队（但可以用 subagents）
2. **Lead 角色固定**：创建团队后，Lead 不能转让给 teammate
3. **不支持会话恢复**：关闭后无法用 `/resume` 恢复整个团队
4. **任务状态可能延迟**：Task List 的同步不是实时的，可能有短暂延迟
5. **单团队限制**：一个 Lead 会话只能管理一个团队
