# 第十七章 Subagent vs Agent Teams 选择指南

## 60 秒决策框架

当你面对一个需要并行处理的任务时，问自己三个问题：

```
Q1: 并行的任务之间需要互相通信吗？
    │
    ├─ 不需要 → Q2
    │
    └─ 需要 → Agent Teams
    
Q2: 任务需要独立的 Git 分支吗？
    │
    ├─ 需要 → Git Worktrees
    │
    └─ 不需要 → Q3

Q3: 任务的输出是否需要隔离（避免污染主 agent 上下文）？
    │
    ├─ 需要隔离 → Subagents
    │
    └─ 不需要 → /batch 或单 agent 循环
```

三个问题，60 秒决策。

| 维度 | Subagents | Agent Teams | Worktrees | /batch |
|------|-----------|-------------|-----------|--------|
| **通信** | 只向主 agent | 互相���信 | 完全独立 | 完全独立 |
| **协调** | 主 agent 管理 | 共享任务列表 | 手动 | 无 |

| **启用状态** | ✅ 默认 | ❌ 需启用 | ✅ 默认 | ✅ 默认 |
| **最佳团队规模** | 1-5 | 3-5 | 2-4 | 不限 |
| **学习成本** | 低 | 中-高 | 低 | 低 |

## 场景决策矩阵

| 场景 | 首选 | 原因 |
|------|------|------|
| 并行审查 PR 的三个维度 | Agent Teams | 审查发现需要交叉验证 |
| 同时搜索 5 个模块的代码模式 | Subagents | 搜索是独立的，结果汇总即可 |
| 同时开发 feature-A 和 feature-B | Worktrees | 完全独立的功能分支 |
| 批量修改 20 个配置文件 | /batch | 简单重复操作 |
| 竞争假设调试 | Agent Teams | 假设需要互相验证和挑战 |
| 隔离探索（读大量代码后给结论） | Subagents | 防止大量代码污染主上下文 |
| 大型重构 + 日常开发并行 | Worktrees | 长期任务需要隔离 |
| 新模块开发（前端+后端+测试） | Agent Teams | 跨层需要接口协调 |

## 混合模式进阶

真实场景中，你经常需要组合使用：

### 模式一：Agent Teams + Subagents

```
Team Lead（Delegate Mode）
├── teammate-A（前端）
│   └── 用 subagent 做组件性能基准测试
├── teammate-B（后端）
│   └── 用 subagent 做 API 压力测试
└── teammate-C（测试）
    └── 用 subagent 生成测试数据
```

每个 teammate 用自己的 subagent 做局部的隔离任务，teammates 之间通过 Mailbox 协调。

### 模式二：Worktrees + Agent Teams

```
Worktree 1：稳定版维护
└── 单个 Claude Code（处理 hotfix）

Worktree 2：新版本开发
└── Agent Teams（3 人团队开发新功能）
```

### 模式三：渐进式升级

从简单到复杂，逐步升级你的多代理策略：

```
阶段 1：单 agent
    ↓ 当任务太大时
阶段 2：Subagents（简单并行）
    ↓ 当任务需要协调时
阶段 3��Agent Teams（协作并行）
    ↓ 当需要长期隔离时
阶段 4：Worktrees + Agent Teams（完全方案）
```

## 成本效益分析

假设一个任务单 agent 需要 4 小时、成本 $10：

| 方案 | 预计耗时 | 预计成本 | 适用条件 |
|------|----------|----------|----------|
| 单 agent | 4h | $10 | 任务不可并行 |
| 3 个 Subagents | 1.5h | $15-20 | 任务可拆分，不需通信 |
| 3 人 Agent Teams | 1.5h | $30-40 | 需要通信协调 |
| 3 个 Worktrees | 1.5h | $30 | 完全独立的分支工作 |

**何时值得多花钱**：
- 时间价值高于 token 成本时
- 任务复杂度使单 agent 效果很差时（上下文溢出）
- 任务有天然的并行结构时

**何时不值得**：
- 简单的线性任务
- 预算紧张
- 任务之间有强依赖（并行度低）

## 推荐的学习路径

如果你是多代理新手，建议按以下顺���学习和实践：

1. **第一周**：2 个 subagent 的只读研究任务（零风险）
2. **第二周**：3 个 subagent 的代码审查（练习并行）
3. **第三周**：2 个 worktrees 并行开发小功能（练习隔离）
4. **第四��**：3 人 Agent Teams 做一次代码审查（练习通信）
5. **之后**：Delegate Mode + Plan → Team 做中等规模的功能开发

每一步都在上一步的基础上增加一个新维度。不要一开始就上 5 人 Agent Teams + Delegate Mode + Plan Approval——那是专家配置。
