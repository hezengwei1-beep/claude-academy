# 第十五章 Subagents 子代理实战

## Subagents 是什么

Subagents 是 Claude Code 内置的并行工作机制。与 Agent Teams 不同，Subagents 是**层级关系**：主 agent 创建 subagent，subagent 完成后向主 agent 汇报结果。Subagents 之间不能互相通信。

**类比**：
- Agent Teams = 同事间的协作会议（平等讨论）
- Subagents = 你派出去的调研助理（单独汇报给你）

## 核心特性

| 特�� | 说明 |
|------|------|
| 启用状态 | 默认可用，无需额外配置 |
| 上下文隔离 | 每个 subagent 有独立的上下文 |
| 通信方向 | 单向：subagent → 主 agent |
| 模型选择 | 可单独指定（如用 haiku 降成本） |
| 嵌套 | subagent 不能创建子 subagent |

## 创建 Subagent 的四种方式

### 方式一：/agents 管理界面

```
/agents
```

打开 agents 管理界面，可以查看、创建和管理 subagents。

### 方式二：自然语言

```
用一个 subagent 去调查 src/auth/ 模块的所有数据库查询，
统计查询次数和类型，结果汇总给我。
```

主 agent 会自动创建一个 subagent 来执行这个任务。

### 方式三：@-mention

```
@code-reviewer 请审查 src/auth/middleware.ts 的最近改动
```

如果你已经定义了名为 `code-reviewer` 的 agent，@-mention 会强制使用它。

### 方式四：/fork 分支会话

```
/fork 并行测试三种不同的排序算法实现
```

需要先启用：
```bash
export CLAUDE_CODE_FORK_SUBAGENT=1
```

## 并行模式

### 后台并行

多个 subagent 同时在后台工作：

```
请用 3 个 subagent 并行调查以下问题：
1. subagent-1：分析 src/api/ 目录的 API 响应时间瓶颈
2. subagent-2：检查 src/db/ 的查询效率
3. subagent-3：审查 src/cache/ 的缓���命中率

各自独立工作，完成后汇报给我。
```

三个 subagent 同时启动，各自完成后结果回到主 agent。

### 链式调用

一�� subagent 的结果作为下一个的��入：

```
先用 code-analyzer subagent 找出性能热点，
然后用 optimizer subagent 基于分析结果提出优化方案。
```

### 前台阻塞

等待 subagent 完成后再继续：

```
等 subagent 完成代码审查后，把结果整合到 PR 评论中。
```

## 模型选择优化

不是所有 subagent 任务都需要最强的模型。简单的搜索和统计任务可以用更便宜的模型：

```bash
# 全局设置 subagent 默认模型
export CLAUDE_CODE_SUBAGENT_MODEL=haiku
```

| 任务类型 | 推荐模型 | 原因 |
|----------|----------|------|
| 深度代码审查 | opus/sonnet | 需要理解复杂逻辑 |
| 文件搜索和统计 | haiku | 简单任务，haiku 足够 |
| 数据提取 | haiku | 模式匹配任务 |
| 架构分析 | sonnet | 需要理解设计模式 |
| Bug 定位 | sonnet/opus | 需要推理能力 |

## 最佳实践

### 1. 用 subagent 隔离"探索型"任务

主 agent 的上下文是宝贵的。如果一个任务需要读取大量文件但你只需要一个结论，用 subagent 隔离它：

```
# 差：在主 agent 中直接读
读取 src/ 下所有 .ts 文件，统计代码行数

# 好：用 subagent 隔离
用一个 subagent 统计 src/ 下所有 .ts 文件的代码行数，
只需要告诉我最终数字。
```

subagent 读了 100 个文件的内容不会污染主 agent 的上下文——只有最终的统计结果会回来。

### 2. 明确 subagent 的输出格式

```
# 差
用 subagent 看看代码有什么问题

# 好
用 subagent 审查 src/api/routes.ts，
输出格式：每个问题一行，"[严重度] 行号: 问题描述"
最多报告 5 个最严重的问题。
```

### 3. 不要用 subagent 做需要协调的任务

如果两个任务之间有依赖或需要互相通信，用 Agent Teams 而不是 subagent。Subagent 之间无法通信，强行用会导致信息割裂。

## Subagent vs Agent Teams 快速判断

问自己一个问题：**这些并行任务需要互相"说话"吗？**

- 不需要 → Subagents
- 需要 → Agent Teams

就这么简单。
