# 第四章 Agent 设计模式

## 为什么需要设计模式

Agent 的 ReAct 循环是底层机制。但不同任务需要不同的"策略"来引导 Agent 如何思考和行动。这些策略就是设计模式。

## 模式 1：ReAct（推理 + 行动）

最基础的模式，也是 Claude Code 的默认模式。

```
思考 → 行动 → 观察 → 思考 → 行动 → 观察 → ...
```

**适合**：目标明确、步骤可推导的任务

**示例**：
```
修复 src/auth/login.ts 中的认证 bug
→ [思考] 需要先看代码
→ [Read] src/auth/login.ts
→ [观察] 第 42 行有问题
→ [思考] 需要修改
→ [Edit] 修复第 42 行
→ [思考] 需要验证
→ [Bash] npm test
→ [观察] 测试通过
→ 完成
```

**在 Claude Code 中使用**：这是默认行为，不需要特殊配置。

## 模式 2：Plan-Execute（先规划后执行）

先制定完整计划，然后按计划执行。

```
分析 → 制定计划 → [按计划逐步执行] → 验证
```

**适合**：大型任务、多文件改动、需要协调的工作

**在 Claude Code 中使用**：
```
/plan

重构用户认证模块，从 Session 迁移到 JWT。
先出方案不要动手。
```

Claude 会输出：
```
## 迁移计划

### Phase 1: 基础设施
1. 安装 jsonwebtoken 包
2. 创建 JWT 工具函数（sign/verify）
3. 更新 .env 添加 JWT_SECRET

### Phase 2: 迁移
4. 改写 auth middleware
5. 更新 login handler
6. 更新 logout handler

### Phase 3: 清理
7. 删除 session 相关代码
8. 更新测试
9. 更新文档

是否按此计划执行？
```

确认后再逐步执行。

**与 Agent Teams 结合**：
```
计划确认后，创建 3 人团队：
- Phase 1 由 infra-dev 执行
- Phase 2 由 auth-dev 执行（依赖 Phase 1）
- Phase 3 由 test-dev 执行（部分可并行）
```

## 模式 3：Reflection（反思）

Agent 做完后自我审查，发现问题再修正。

```
执行 → 自查 → 发现问题 → 修正 → 再自查 → 通过
```

**适合**：质量要求高的任务（安全审计、重要文档）

**在 Claude Code 中使用**：
```
帮我写一个支付处理函数。
写完后，自己从安全角度审查一遍，找出潜在问题并修复。
```

Claude 会：
1. 先写代码
2. 然后切换到"审查者"视角
3. 找出自己代码中的问题
4. 修复问题
5. 再审查一次确认

**高级用法——用 Subagent 做 Reflection**：
```
用一个 subagent 来审查你刚才写的代码。
subagent 的角色是安全专家，专找安全漏洞。
基于它的发现修复问题。
```

这样"写代码的 Agent"和"审查代码的 Agent"是不同的实例，不会有自我偏见。

## 模式 4：Tool-First（工具驱动）

让工具的输出引导 Agent 的下一步决策。

```
调用工具 → 基于工具输出决定下一步 → 调用下一个工具 → ...
```

**适合**：数据驱动的任务、需要外部信息的任务

**示例**：
```
1. 查数据库 → 发现异常数据
2. 基于异常查代码 → 找到相关逻辑
3. 分析代码 → 发现 bug
4. 修复 → 验证
```

**与 MCP 结合**：
```
1. [MCP: PostgreSQL] 查询昨天的错误日志
2. [MCP: GitHub] 查找相关的 Issues
3. [MCP: Slack] 查看团队讨论
4. 综合分析根因
5. [MCP: GitHub] 创建修复 Issue
```

## 模式 5：Divide-Conquer（分治）

大任务拆成小任务，各自解决后合并。

```
大任务 → 拆分为 N 个子任务 → 并行处理 → 合并结果
```

**适合**：可并行化的大型任务

**在 Claude Code 中使用**：
```
分析整个 src/ 目录的代码质量：

用 5 个 subagent 各负责一个子目录：
- subagent-1: src/api/
- subagent-2: src/services/
- subagent-3: src/models/
- subagent-4: src/utils/
- subagent-5: src/middleware/

各自分析后汇总成整体报告。
```

## 模式 6：Debate（辩论）

多个 Agent 持不同立场讨论，通过辩论得出更好的结论。

```
Agent A（支持方案 1）↔ Agent B（支持方案 2）→ 综合最优方案
```

**在 Agent Teams 中使用**：
```
创建一个 3 人架构讨论团队：

1. advocate-microservice
   - 立场：主张用微服务架构
   - 任务：分析微服务的优势，回应对方的质疑

2. advocate-monolith
   - 立场：主张用单体架构
   - 任务：分析单体的优势，回应对方的质疑

3. moderator
   - 立场：中立
   - 任务：引导讨论，最后综合两方观点给出建议

让他们讨论我们的项目应该用什么架构。
```

这个模式特别适合**技术决策**——单 Agent 容易有偏见，多 Agent 辩论能暴露更多考虑因素。

## 模式选择速查

| 任务类型 | 推荐模式 | 在 Claude 中如何实现 |
|----------|---------|-------------------|
| 简单修复 | ReAct | 默认行为 |
| 大型重构 | Plan-Execute | /plan → Agent Teams |
| 安全审计 | Reflection | Subagent 交叉审查 |
| 数据分析 | Tool-First | MCP + 数据库 |
| 代码库全面分析 | Divide-Conquer | 多个 Subagent 并行 |
| 技术决策 | Debate | Agent Teams 辩论 |
| 日常开发 | ReAct | 默认 |
